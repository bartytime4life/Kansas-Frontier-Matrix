## Purpose

This README defines a non-authoritative boundary for reusable Archaeology-specific identity support while the implementation home remains unresolved.

A future accepted implementation may:

- validate explicitly typed, caller-supplied identity inputs;
- preserve source-native identifiers without treating them as KFM identifiers;
- construct deterministic candidate identity material under an accepted, versioned profile;
- produce proposed crosswalk, possible-duplicate, same-as, split, merge, supersession, and correction relations for external review;
- compare candidates inside an explicit caller-supplied scope and return explanations without deciding sameness;
- preserve candidate/site distinction, object role, source role, temporal scope, evidence, review, rights, sensitivity, lineage, and release references;
- accept canonical bytes or declared digests from the owning canonicalization/hashing boundary instead of inventing them locally;
- support opaque identifiers for already-authorized public derivative objects without linking them to restricted originals;
- return finite typed results and stable reason codes with no hidden I/O;
- support synthetic, sanitized, no-network tests and cross-implementation parity vectors.

It must not define universal identifier grammar, decide object sameness, confirm sites, mutate canonical registries, merge or split records, infer sensitive relationships, downgrade source sensitivity, create EvidenceBundles, satisfy review, select policy, approve release, serve public resolution, or expose controlled identifiers or locations.

## Bounded context and anti-collapse rules

| Term | Meaning in this boundary |
|---|---|
| Source-native ID | Identifier asserted by a named source in a declared source role. It is preserved, not promoted into KFM authority. |
| CandidateFeature ID | Identity of a candidate observation or feature. It must remain distinct from a confirmed ArchaeologicalSite identity. |
| ArchaeologicalSite ID | Stable identity of a governed site object after an external evidence/review transition; helper output cannot create it. |
| DomainFeatureIdentity | Draft identity/crosswalk semantic object for aliases and lineage; current schema does not enforce its rich semantics. |
| Identity profile | Named, versioned rules for inputs, normalization, canonicalization, namespace, digest, scope, and comparison. No accepted registry was established. |
| Identity seed | Explicit typed content from which a candidate identifier is derived under a profile. It excludes environment and transport entropy. |
| Alias/crosswalk | Governed relationship between identifiers. It is not automatically equality or public resolvability. |
| Match candidate | Local comparison result with reasons and scope. It is not a same-entity decision. |
| Public derivative ID | Identifier for a separately governed released derivative object, not an alias that reveals a restricted original. |

Disallowed collapses:

```text
valid syntax -> existing object
same normalized string -> same entity
same digest -> same site
source-native ID -> canonical KFM ID
CandidateFeature ID -> ArchaeologicalSite ID
similarity score -> duplicate decision
possible-same-as -> same-as
same-as proposal -> merge
merge proposal -> registry mutation
alias -> owl:sameAs
internal ID -> public ID
opaque hash -> safe public identifier
public derivative ID -> authorization to resolve the source
EvidenceRef -> evidence closure
review reference -> completed review or consent
determinism -> correctness, uniqueness, or policy approval
successful helper test -> lifecycle promotion or release
```

Identity must be content-derived under an explicit profile, never from ambient file paths, process IDs, run IDs, hostnames, retrieval time, current time, random values, environment variables, mutable labels, transport URLs, log order, or serializer accidents unless a contract explicitly defines the value as identity-bearing domain content.

## Identity profile separation

One universal identifier function would erase the very distinctions identity must preserve. Profiles must be explicit and versioned.

| Profile family | Required separation |
|---|---|
| Source record | Source authority, dataset/version, source role, native ID, and logical temporal scope. |
| Candidate feature | Candidate type, observation scope, method/sensor role, and evidence context; never site status. |
| Archaeological site | Governed site object after accepted transition; naming conflict must be resolved first. |
| Survey/event/context | Event or context identity remains distinct from the feature or site observed. |
| Artifact/collection | Object, collection, accession, and provenience roles remain distinct. |
| Evidence/support | EvidenceRef and EvidenceBundle identities remain evidence-owned. |
| Transform derivative | A sensitivity/generalization derivative receives a new identity and lineage; it is not an alias of exact controlled geometry. |
| Public release artifact | Release/profile/audience-scoped artifact identity remains distinct from canonical and internal identity. |

### Composition constraints

The draft identity model proposes a basis equivalent to:

```text
source identity + object role + temporal scope + normalized content digest
```

The basis is useful doctrine, not an accepted field-level recipe. A future profile must define:

- required and prohibited fields;
- Unicode, whitespace, case, punctuation, number, null, list, and map rules;
- canonical byte encoding and hash algorithm/version;
- namespace, kind, prefix, length, and character restrictions;
- logical time kind, calendar, timezone, and EDTF handling when time participates;
- collision scope and response;
- algorithm agility and migration behavior;
- domain and public/restricted audience scope;
- test vectors that can be reconstructed offline.

Source-controlled text is inert data. It may be normalized only by the selected profile and must never be interpreted as instructions that change field selection, profile selection, code path, policy, or identity output.

### Required behavior

- Pure and deterministic for identical accepted inputs and profile version.
- No network, database, filesystem, registry, clock, environment, random, or hidden cache access.
- No fallback profile or silent repair.

- No partial success that drops source, time, evidence, sensitivity, or lineage distinctions.

- `candidates` means suggestions for external decision, never a changed registry or accepted relation.
- Errors fail closed and reveal no controlled input values.

Stable reason codes, error types, size limits, recursion limits, comparison limits, and time bounds require acceptance before implementation.

## Matching, merge, split, correction, and lineage

Identity lifecycle decisions are external governance events. Helpers may propose records but must not apply them.

| Event | Helper may | Helper must not |
|---|---|---|
| Possible duplicate | Return scoped candidates, scores, features used, excluded sensitive features, and deterministic reasons. | Assert equality or mutate either record. |
| Same-as | Construct a review candidate referencing both identities, profile, evidence, and scope. | Treat semantic similarity or a matching native ID as proof. |
| Merge | Validate or construct an externally authorized merge proposal. | Choose survivor, move evidence, rewrite references, or delete identities. |
| Split | Validate or construct an externally authorized split proposal. | Invent descendants, repartition evidence, or conceal uncertainty. |
| Correction | Derive a new candidate under accepted correction input and preserve predecessor relation. | Rewrite history or reuse a retired ID. |
| Supersession/tombstone | Format an accepted lineage relation. | Decide retirement, public withdrawal, or resolution behavior. |

Minimum lifecycle invariants:

- identifiers are never silently recycled;
- source-native identifiers and original assertions remain inspectable under their access rules;
- merge and split decisions record decision authority, evidence, review, time, scope, and reversibility;
- old identifiers resolve only according to explicit policy and audience, never by an implicit redirect;
- correction creates traceable lineage and triggers downstream impact review;
- rollback restores governed state and mappings without erasing decision history;
- public artifacts, graph edges, indexes, tiles, citations, caches, and AI-derived summaries are treated as downstream consumers requiring correction or withdrawal review.

## Sensitivity, privacy, and threat model

Archaeology identifiers are disclosure surfaces. Even an opaque-looking value can reveal existence, location, source, collection, chronology, relationship, reviewer activity, or release history through its inputs or use.

### Required sensitivity rules

- Never derive a public identifier directly from exact coordinates, parcel IDs, site numbers, accession values, source URLs, owner/custodian names, access routes, protected cultural terms, or other low-entropy controlled fields.
- Hashing is not redaction. An attacker can enumerate low-entropy inputs and compare digests.
- Do not expose sequential allocation, source prefixes, geographic codes, type prefixes, timestamps, shard IDs, sequence gaps, collision counters, or error distinctions that reveal controlled facts.
- Keep internal, partner, reviewer, and public identifier profiles separate.
- A public identifier belongs to a separately governed derivative or released artifact. It must not be a reversible or correlatable alias for a restricted exact object unless explicit policy authorizes that relationship.
- A sensitivity transform creates a new derivative identity with lineage; it does not downgrade or rename the controlled original.
- Crosswalk resolution is audience-checked and purpose-bound outside this helper.
- Withhold output when the authorized profile cannot prevent disclosure or linkage.

### Threats to test

| Threat | Required posture |
|---|---|
| Dictionary/enumeration attack | Exclude low-entropy sensitive inputs from public derivation; test realistic enumeration. |
| Cross-release correlation | Rotate or separate audience/release scopes where policy requires; test cumulative linkage. |
| Graph linkage | Treat aliases, shared prefixes, source roles, and stable join keys as potentially sensitive. |
| Canonicalization drift | Pin profile/version and parity vectors; fail closed on unknown versions. |
| Unicode confusables and spoofing | Define normalization explicitly; display original and normalized forms safely to reviewers. |
| Digest collision | Preserve full algorithm/version and collision scope; escalate rather than auto-resolve. |
| Identifier injection | Constrain length/characters; keep source text inert; escape output by downstream context. |
| Replay/stale mapping | Bind decisions to version, lifecycle state, and correction lineage outside the helper. |
| Timing and error oracle | Bound work and use non-sensitive reason families. |
| Log/telemetry leakage | Log only approved metadata and opaque reason codes. |
| Dependency compromise | Minimize dependencies, pin/review them, and test reproducible builds before admission. |

## Trust membrane and authority flow

```text
RAW source assertions
  -> governed WORK / QUARANTINE records
  -> typed identity-profile input
  -> pure helper result or match/relation candidate
  -> independent contract/schema validation
  -> evidence + cultural/steward review + policy
  -> authorized identity/lifecycle decision and registry write
  -> CATALOG / TRIPLET projections
  -> separately governed public derivative and ReleaseManifest
  -> PUBLISHED resolution/delivery
```

The helper can occupy only the pure-computation step. It may consume references, but it cannot verify the records behind them, cross the trust membrane, write lifecycle state, project graphs, or release an identifier.

Public catalogs, graph projections, indexes, map layers, search results, and AI answers are downstream views. They do not become identity authority by repeating or linking a helper output.

## Validation, testing, and admission

The current repository does not establish package-specific executable coverage for this boundary. Tests should extend the accepted Archaeology test topology after the implementation home is resolved.

### Minimum future suites

| Suite | Required cases |
|---|---|
| Golden/parity vectors | Same accepted input produces identical canonical bytes and identifier across runs, platforms, languages, and serializer implementations. |
| Profile boundaries | Unknown profile/version/kind rejected; no fallback; source role and object role affect identity as specified. |
| Text/numeric forms | Unicode normalization, confusables, case, whitespace, punctuation, ordering, integers/decimals, null/missing, arrays, and maps. |
| Temporal scope | Time kinds remain distinct; EDTF/calendar/timezone behavior explicit; retrieval or run time excluded. |
| Domain separation | CandidateFeature never equals ArchaeologicalSite; `Site` conflict cannot be silently collapsed; artifact/context/event/evidence kinds remain separate. |
| Matching | Deterministic candidate ordering, explicit scope, ambiguous ties, no auto-merge, and reason-code stability. |
| Lifecycle | Merge/split/correction/supersession proposal validation, tombstones, no ID reuse, replay, and rollback. |
| Sensitivity | Enumeration, linkage, source-prefix, sequence, timing, error, multi-release, and public/internal crosswalk attacks. |
| Security | Oversized/deep input, Unicode spoofing, path/URL/control characters, injection strings, denial-of-service bounds, and no secret or payload echo. |
| Hash agility | Unknown algorithm rejection, full tag preservation, dual-read migration vectors, collision escalation, and no silent truncation. |
| Purity | No network, clock, filesystem, environment, random source, registry write, telemetry payload, or hidden cache dependency. |

Fixtures must be synthetic or formally sanitized. Real coordinates, site IDs, collection identifiers, protected names, source access tokens, reviewer identities, or reconstructive relationships do not belong in ordinary repository fixtures.

1. implementation-home and shared/domain responsibility decision accepted;
2. owners, CODEOWNERS, review authorities, and threat model accepted;
3. object/profile vocabulary and `Site` naming conflict resolved;
4. semantic contracts and closed schemas expanded together;
5. canonicalization/hash algorithm, wire form, namespace, collision scope, and agility plan accepted;
6. internal/partner/public identifier and resolution policies accepted;
7. merge/split/correction/tombstone authority and audit records accepted;
8. package API, limits, typed errors, dependencies, runtime, and export surface accepted;
9. golden, parity, lifecycle, privacy, adversarial, and no-network tests executable;
10. validator and CI enforcement implemented without bypassing the Archaeology readiness hold;
11. consumer inventory, migration, correction cascade, rollback drill, and release criteria reviewed;
12. cultural, sovereignty, privacy, security, evidence, policy, and release stewards approve their boundaries.

## Compatibility, migration, and rollback

Identity behavior is compatibility-sensitive. A profile, canonicalization, namespace, or object-kind change can alter every derived identifier and downstream reference.

Before changing accepted behavior:

- inventory producers, consumers, persisted IDs, aliases, crosswalks, evidence references, graph edges, indexes, tiles, citations, releases, and caches;
- publish old/new golden vectors and an impact comparison;
- assign a new profile or algorithm version rather than mutating old semantics in place;
- define dual-read/single-write behavior, ambiguity handling, and a bounded compatibility window;
- prohibit silent backfill, suffixing, retargeting, alias expansion, and public redirect changes;
- record migration, correction, and rollback decisions through accepted governed artifacts;
- rehearse rollback with evidence/reference preservation and no ID reuse;
- verify public withdrawal and cache invalidation for affected released artifacts;
- retain prior profile definitions and decision history for offline reconstruction.

Documentation rollback reverts this README and its generated receipt. Software or data rollback requires the owning package, registry, pipeline, and release procedures; reverting documentation does not undo an identity decision or published artifact.

## Maintainer note

Resolve the implementation home before adding code. Keep helpers pure, typed, profile-bound, deterministic, non-authoritative, no-network, and safe for a sensitive domain. Preserve source, object-role, temporal, evidence, review, rights, sensitivity, lineage, and release distinctions. Do not turn normalization into sameness, a digest into proof, a match into a merge, an alias into public resolution, or a public identifier into a link to a restricted site.
