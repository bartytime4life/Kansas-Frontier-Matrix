## Source admission rules

A production source file may enter this envelope only when its responsibility is reusable redaction mechanics and its governance burden is explicit.

### Admission questions

1. **Reusability:** Is the behavior required by more than one governed caller or a clearly shared contract?
2. **Authority:** Does the file execute supplied obligations rather than decide policy, evidence, rights, sensitivity, review, or release?
3. **Input:** Are all material inputs explicit, inspectable, versioned, and caller supplied?
4. **Determinism:** Is deterministic behavior specified where promised, including normalization, geometry, randomness, time, and serialization?
5. **Failure:** Are missing, stale, malformed, unsupported, or conflicted inputs represented by explicit fail-closed outcomes?
6. **Sensitivity:** Can implementation, logs, errors, examples, and fixtures avoid protected data and reconstruction secrets?
7. **Testing:** Do valid, invalid, negative, replay, security, and no-network tests exist?
8. **Placement:** Would the file create a parallel contract, schema, policy, registry, receipt, proof, release, data, API, or UI authority?
9. **Compatibility:** Is the API surface intentional, minimal, versioned, and reversible?
10. **Consumer:** Is there a named governed consumer and integration test, or is the file premature framework code?

A file failing an admission question remains outside the supported source envelope until the gap is resolved. One-off transformation jobs belong in `tools/` or `pipelines/`; source-specific fetch/admission belongs in `connectors/`.

### No-hidden-fetch invariant

Source code must not fill missing policy, evidence, rights, sensitivity, geometry, release, review, or profile context from:

- RAW, WORK, QUARANTINE, or unpublished candidate stores;
- source systems or connectors;
- UI/browser state;
- mutable current-release or “latest” pointers;
- operator memory or process-global configuration;
- vector search or generated language;
- model output;
- ambient credentials or secrets.

Missing authority remains missing. Return an explicit negative state rather than inventing context.

## Transform and profile boundary

A redaction profile is policy/governance input. It is not authored, selected, activated, or retired by package source.

### File presence is not activation

The following must not activate a profile:

- being the newest file;
- a directory name;
- a mutable `latest` alias;
- a symlink;
- successful parsing;
- schema validity alone;
- an environment-variable fallback;
- an API or UI request parameter;
- model output;
- a default embedded in package source.

A future binding should identify profile id/version/digest, status, policy basis, contract/schema versions, allowed transform class, parameter visibility, deterministic/replay rules, activation record, supersession state, and failure posture.

### Transform rules

A supported transform must:

1. operate only on the explicit target;
2. receive an accepted profile and obligations;
3. validate profile identity/version/status before execution;
4. distinguish no-change, transformed, withheld, unsupported, denied-by-policy-input, and execution-failure semantics through accepted contracts;
5. preserve non-sensitive lineage and replay context;
6. avoid leaking protected input through output, metadata, exceptions, logs, timing, or fixtures;
7. produce no public or lifecycle side effect;
8. never treat process success as release approval.

Profile parameter secrecy and verifier replayability may be in tension. The accepted contract must define which parameters/digests are safe to persist and which require restricted references.

## Outcome vocabularies and fail-closed behavior

The current package defines no runtime outcome type. The prior source README named:

```text
SAFE | REDACTED | GENERALIZED | WITHHELD | DENIED | HOLD | ABSTAIN | INVALID
```

Those values may be useful as design language, but they are **not an accepted package API**.

The draft redaction-determinism standard uses verifier outcomes:

```text
ANSWER | ABSTAIN | DENY | ERROR
```

Other validators and release workflows may use separate vocabularies. These layers must not be silently collapsed.

| Layer | Purpose | Status |
|---|---|---:|
| Transform-local result | Describe mechanical change/no-change/no-output/error. | **UNRATIFIED** |
| Policy decision | Select obligation or block progress. | External authority; exact binding NEEDS VERIFICATION. |

| Runtime envelope | Public/internal transport of governed finite result. | Must use accepted runtime contracts. |
| Release state | Candidate/reviewed/released/withdrawn/etc. | Separate release vocabulary. |

Required distinctions:

- missing support is not execution error;
- policy denial is not malformed input;
- withheld output is not successful public release;
- transformed output is not evidence truth;
- review hold is not implicit allow;
- verifier `ANSWER` is not publication approval.

A versioned normalization contract is required before source code maps any local result into policy, runtime, receipt, or release outcomes.

- make `__init__.py` explicit and minimal;
- test source-tree and installed-artifact API parity.

Breaking changes may include:

- input/output field or semantic changes;
- outcome vocabulary or reason-code changes;
- profile-binding rules;
- canonicalization, hash, geometry, random, or time changes;
- leakage/visibility changes;
- exception and validation behavior;
- dependency-driven output differences.

### Deprecation

A deprecation should identify:

- old and replacement symbols/contracts/profiles;
- affected consumers;
- warning audience and safe message;
- support window;
- migration tool or adapter;
- receipt/replay consequences;
- removal gate;
- rollback target.

Do not use deprecation warnings that expose sensitive input or profile secrets.

### Correction and revocation

If a transform implementation, profile, canonicalization rule, or dependency is found unsafe or incorrect:

1. disable affected consumer/profile bindings;
2. preserve affected execution/receipt/release lineage;
3. identify impacted candidates, receipts, proofs, releases, exports, and public artifacts;
4. quarantine or withdraw where required;
5. issue correction/withdrawal records through owning workflows;
6. repair implementation and fixtures;
7. re-run validation/replay;
8. re-release only after governed gates pass.

Changing source code does not silently rewrite historical receipts or released artifacts.

## Rollback

### Documentation rollback

Revert the documentation commit or close the unmerged review pull request. Preserve history; do not reset or force-rewrite shared history. Re-run documentation, link, receipt, and scope checks.

### Future software rollback expectations

A source implementation change must identify:

- prior package and implementation version;
- affected exports and consumers;
- profile, policy, contract, schema, canonicalization, dependency, and validator versions;
- affected derivative candidates, receipts, proofs, releases, exports, and public artifacts;
- safe disable and restoration order;
- data/public correction obligations;
- post-rollback validation and replay;
- supersession and deprecation handling.

Rolling back source code does not automatically roll back policy decisions, receipts, proofs, releases, or public artifacts. Preserve lineage and use transparent correction, withdrawal, and rollback processes.
