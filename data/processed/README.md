## üìö Overview

The `data/processed/` directory serves as the **canonical layer** for all data products derived from raw sources after validation, transformation, and enrichment by the KFM ETL pipelines.  
All files adhere to open, machine-readable formats (GeoJSON, Cloud-Optimized GeoTIFF, CSV, JSON-LD) with complete metadata and provenance links.

### Core Functions

- **Normalization:** Converts raw and intermediate data into interoperable standards (e.g., GeoTIFF for rasters, GeoJSON for vectors).  
- **Governance Readiness:** Includes checksum, lineage, and validation reports compliant with FAIR+CARE and Master Coder Protocol.  
- **Discoverability:** Indexed in the [STAC Catalog](../../data/stac/catalog.json) for spatial and temporal search.  
- **Interoperability:** Fully DCAT-3 and CIDOC CRM aligned for cross-domain data exchange.

## üßÆ Validation and Provenance

Each dataset is accompanied by validation artifacts:

- **Checksum Verification:** SHA-256 recorded in manifest files.  
- **Schema Validation:** JSON Schema validation for metadata and data structure.

## üîó Linkages and Semantic Integration

Processed datasets are semantically linked through the knowledge graph and ontologies:

- **Spatial Entities:** `geo:hasGeometry` ‚Üí Links to places or features.  
- **Temporal Extents:** `time:hasBeginning` / `time:hasEnd` ‚Üí Connects with OWL-Time intervals.  
- **Provenance:** `prov:wasGeneratedBy` ‚Üí Pipeline or ETL process identifiers.  
- **Cultural Context:** `crm:E7_Activity` / `crm:P7_took_place_at` for events and treaty layers.  

All metadata includes a JSON-LD context for direct RDF export compatibility.
