## ğŸ“˜ Overview

This directory is the **authoritative, dataset-local STAC home** for Surficial Geology.

Intended contents:

- **STAC Collection JSON** describing the Surficial Geology dataset at the collection level.
- **STAC Item JSON** describing publishable assets (e.g., vector products, tilesets, styles, legends),
  with each Item linking to its actual assets stored under:
  - `data/surficial-geology/outputs/tiles/`
  - `data/surficial-geology/outputs/vectors/`
  - `data/surficial-geology/outputs/metadata/`
- **Deterministic, replayable outputs**: items should be generated by a pipeline/run config whenever possible,
  not manually edited post-hoc.

Non-goals:

- Do **not** store large binaries here (tiles, geopackages, geotiffs, etc.).
- Do **not** duplicate raw sources here (those belong in `data/surficial-geology/raw/`).
- Do **not** embed secrets, credentials, or sensitive locations at high precision.

## ğŸ“¦ Data & Metadata

### Object types

- **Collection**
  - Use a stable `id` (recommended: `surficial-geology` or `kfm-surficial-geology`).
  - Provide dataset-level description, spatial/temporal extent, license, keywords, and links.

- **Item**
  - One Item per publishable unit (e.g., statewide tileset, vector package, or a defined thematic layer).
  - Items MUST link to their assets (tiles/vectors/metadata) via `assets.*.href`.

### Naming and IDs

Recommended patterns (choose one and apply consistently):

- Collection:
  - `id: "surficial-geology"`
- Items:
  - `id: "surficial-geology::<asset_kind>::<version_or_run_id>"`
  - Example (pattern only): `surficial-geology::vectors::run_2025-12-14T1530Z`

### Assets

Assets should point to repo-relative locations wherever possible, e.g.:

- `data/surficial-geology/outputs/vectors/...`
- `data/surficial-geology/outputs/tiles/...`
- `data/surficial-geology/outputs/metadata/...`
- `data/surficial-geology/lineage/prov/...` (provenance payloads)
- `data/surficial-geology/lineage/openlineage/events/...` (run events)

Prefer meaningful `roles`, for example:

- `["data"]`, `["metadata"]`, `["visualization"]`, `["provenance"]`

## ğŸŒ STAC, DCAT & PROV Alignment

### STAC

- Treat this directory as a **static STAC catalog** (even if later served via a STAC API).
- Keep core fields complete and consistent:
  - `id`, `type`, `stac_version`
  - `links` (self/parent/collection as appropriate)
  - `extent` (Collection) and `geometry`/`bbox` (Item when spatial)
  - `assets` with stable, valid hrefs

### DCAT

- Collection-level metadata should be sufficient to map to a DCAT `dcat:Dataset`:
  - title/description, keywords/themes, publisher/source, license, spatial/temporal coverage
- Item assets map naturally to DCAT `dcat:Distribution` entries.

### PROV-O / Lineage

- Every Item should be able to point to the run/activity that generated it via:
  - an asset or link to `data/surficial-geology/lineage/prov/`
  - and/or OpenLineage event JSON under `data/surficial-geology/lineage/openlineage/events/`
- If versioning is used, prefer explicit predecessor/successor linkage (by ID and link relations).
