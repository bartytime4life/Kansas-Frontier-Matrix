## Purpose

This directory is the parent test package for Archaeology fixture behavior.

Its durable question is:

> Can tests prove that synthetic Archaeology examples preserve object identity, source role, candidate posture, evidence limits, rights, cultural review, sensitivity, policy, release state, correction lineage, and rollback support—without allowing fixture presence or test success to become authority?

A mature parent suite should prove all of the following:

1. fixture identity is explicitly synthetic, test-local, or reusable under a documented fixture contract;
2. fixture routing selects one appropriate home and does not create parallel payload authority;
3. source-shaped examples remain examples and cannot activate or admit a source;
4. candidate-shaped examples remain candidates and cannot become confirmed archaeological sites by naming, geometry, map rendering, API wording, or generated prose;
5. exact and reverse-engineerable protected detail is absent or represented only by conspicuous safe canaries;
6. source role, knowledge character, object family, units, time kinds, caveats, and review state remain explicit;
7. missing evidence, rights, consent, cultural authority, policy, review, release, correction, or rollback support fails closed;
8. default tests are deterministic and no-network;
9. test outputs use finite, inspectable outcomes and safe reason codes;
10. fixtures, tests, reports, and generated prose remain subordinate to evidence, policy, review, release, correction, and rollback.

This parent does not define Archaeology object meaning, schema shape, policy rules, cultural authority, source authority, or release decisions.

## Three fixture surfaces

The repository contains three legitimate but easily confused fixture-related surfaces.

| Surface | Purpose | May contain | Must not become |
|---|---|---|---|
| `fixtures/domains/archaeology/` | Reusable fixtures shared across multiple tests, validators, examples, or pipelines. | Compact synthetic payloads, valid/invalid cases, expected outputs, reusable canaries. | Source data, site registry, evidence authority, policy, release, or public truth. |
| `tests/fixtures/domains/archaeology/` | Fixtures local to a test area or expectation manifests that should not become cross-cutting. | Small wrappers, path manifests, parametrization data, expected reason-code maps. | Second canonical fixture registry or executable test package. |
| `tests/domains/archaeology/fixtures/` | Executable tests that prove Archaeology fixture behavior. | Test modules, shared test helpers with narrow scope, child test-lane READMEs. | Fixture payload authority, implementation root, source registry, or release authority. |

### Routing questions

Before adding a file, answer:

1. Is this reusable payload data used by more than one test or validator?  
   Use `fixtures/domains/archaeology/`.
2. Is this a tiny wrapper, expectation map, or parametrization manifest owned by one test area?  
   Use `tests/fixtures/domains/archaeology/`.
3. Is this executable logic proving fixture behavior?  
   Use this parent or an accepted child test lane.
4. Is this object meaning, shape, policy, source identity, lifecycle data, trust record, release record, or public output?  
   Use the owning responsibility root, not any fixture surface.

### Duplicate-authority rule

The same fixture ID, payload, expected result, or semantic case must not be independently maintained in more than one fixture home.

A wrapper may reference a reusable fixture. It should not copy and silently fork it.

## Fixture selection and routing contract

### Reusable fixture contract

A reusable fixture should have:

- stable fixture ID;
- synthetic marker;
- object or envelope family;
- source role or candidate posture where applicable;
- sensitivity and public-safety posture;
- expected valid/invalid/denied/abstained/quarantined outcome;
- expected reason code;
- schema/contract/policy version refs when established;
- consumer backlinks;
- correction/supersession metadata where applicable;
- deterministic content hash or version when mature.

### Test-local wrapper contract

A test-local wrapper should additionally state:

- owning test area;
- why it is not reusable;
- referenced reusable fixture ID, if any;
- expected test assertion;
- expiration or cleanup condition;
- no-copy rule.

### Inline test data

Tiny inline examples are acceptable only when:

- they are shorter and clearer than an external fixture;
- they are not reusable;
- they contain no sensitive or plausible real-world detail;
- their expected result is obvious;
- the test documents why they are inline.

### Fixture mutation discipline

A fixture change is a behavior change when it affects expected outcomes.

Update together:

1. fixture payload or wrapper;
2. owning tests;
3. schema/contract/policy refs;
4. expected result and reason code;
5. snapshots or retained reports;
6. fixture and consumer indexes;
7. correction/supersession note;
8. generated receipt for AI-authored changes where required.

Do not silently rewrite a golden fixture to make a failing test pass.

## No-network, security, and sensitive output

### Default no-network posture

Default fixture tests must not:

- resolve DNS or open sockets;
- call source registries, restricted archives, geocoders, map services, tiles, APIs, model providers, release services, or AI runtimes;
- use credentials, tokens, cookies, signed URLs, private endpoints, or developer-local caches;
- write to lifecycle, registry, proof, receipt, release, public, artifact, or cache roots;
- depend on mutable current conditions or external clock state;
- fetch official warnings or operational guidance.

### Untrusted fixture content

Treat fixture strings as untrusted.

Tests should guard against:

- path traversal;
- absolute-path writes;
- symlink escape;
- oversized/decompression-bomb payloads;
- executable content;
- unsafe YAML tags;
- formula injection in tabular exports;
- HTML/script injection;
- log injection;
- prompt injection in AI-bound test envelopes;
- malicious geometry or parser edge cases.

### Protected-output canaries

Use conspicuous fake canaries such as:

```text
fixture://archaeology/safe-canary/001
ARCH_TEST_PROTECTED_VALUE_MUST_NOT_RENDER
MOCK_GEOMETRY_WITHHELD
```

A no-leak test should search:

- returned objects;
- exception messages;
- assertion diffs;
- snapshots;
- JUnit/XML;
- JSON reports;
- logs;
- generated documentation;
- map/API/AI fixture outputs.

### Failure-output minimization

On failure, report:

- fixture ID;
- test ID;
- lane;
- finite outcome;
- safe reason code;
- validator/policy version ref;
- redacted path;
- correction/rollback reference where applicable.

Do not print the full fixture by default.
