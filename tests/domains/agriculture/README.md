## Purpose

`tests/domains/agriculture/` is the Agriculture domain segment inside the canonical `tests/` responsibility root.

Its durable question is:

> Can a bounded Agriculture candidate, cross-domain context record, aggregate, catalog candidate, release candidate, public-surface projection, correction, or rollback scenario be tested deterministically while preserving source identity, domain ownership, spatial and temporal support, evidence closure, rights and sensitivity posture, policy decision, lifecycle state, release state, correction lineage, and rollback readiness?

A mature Agriculture test package should prove at least these boundaries:

1. semantic meaning comes from accepted contracts rather than test-local invention;
2. machine shape comes from the accepted Agriculture or shared schema family;
3. fixtures are synthetic, minimized, deterministic, and public-safe;
4. source identity, source role, source vintage, and support type survive every transform;
5. Agriculture-owned claims remain distinguishable from Soil-, Hydrology-, Atmosphere-, Habitat-, Geology-, Hazards-, and People/Land-owned context;
6. aggregate or modeled material never becomes field-, operator-, farm-, parcel-, or private-party truth by implication;
7. spatial support, temporal support, units, quality flags, no-data state, uncertainty, and caveats remain visible where material;
8. consequential `EvidenceRef` values resolve to adequate `EvidenceBundle` support before answer-like output is accepted;
9. rights, sensitivity, redaction, aggregation, review, and policy obligations fail closed;
10. catalog records, receipts, proofs, schema validity, and test success remain separate from release authority;
11. public API, UI, map, export, search, graph, report, screenshot, and AI carriers cannot bypass release state;
12. correction, supersession, withdrawal, invalidation, and rollback behavior is testable;
13. the default suite performs no live network, model-provider, source-service, database, or public-release operation;
14. failures produce finite, reviewable outcomes rather than silent acceptance.

This lane proves enforceability. It does not create Agriculture truth, source admission, contract meaning, schema authority, policy authority, evidence authority, lifecycle promotion, catalog authority, release approval, or publication.

### Fixture admission rules

A fixture admitted to an Agriculture default test should:

1. be synthetic, minimized, deterministic, public-safe, and obviously marked as non-production;
2. state its case identity, expected outcome, expected reason or failure class, and consuming test;
3. use non-routable identifiers and generalized or fake geometry;

## Required case matrix

A mature Agriculture suite should cover each material profile with the following case classes. Not every class maps to the same runtime envelope; the test must state which layer is being exercised.

| Case class | Required behavior | Typical finite result |
|---|---|---|
| Valid | Meets the bounded contract, schema, source-role, evidence, policy, and release prerequisites for the tested operation. | Test `PASS`; runtime `ANSWER` only when public-use prerequisites also close. |
| Invalid | Malformed, semantically inconsistent, missing required support, or violates an invariant. | Test failure or expected validation error; never silent coercion. |
| Denied | Rights, sensitivity, exact exposure, policy, release, or audience posture prohibits the operation. | `DENY` with safe reason and no protected payload. |
| Abstained | Evidence is absent, stale, conflicted, insufficient, unresolved, or outside requested support. | `ABSTAIN`; no invented value or prose. |
| Held | Governance review, rights clarification, source admission, contract decision, or release decision is pending. | `HOLD` or equivalent bounded review state; active vocabulary `NEEDS VERIFICATION`. |
| Restricted | A safer transformed or limited-audience output may be possible, but requested output is not allowed. | Restrict/generalize/redact obligation or `DENY`; active vocabulary `NEEDS VERIFICATION`. |
| Quarantined | Candidate cannot proceed in the lifecycle because identity, validation, rights, sensitivity, source role, or integrity failed. | Route to `QUARANTINE` with reason and no publication side effect. |
| Error | Tool, schema resolver, policy evaluator, fixture loader, network guard, or runtime failed operationally. | `ERROR`; fail closed and preserve diagnostics without leakage. |
| Correction | Upstream support changed or prior output is materially wrong. | Prior result marked corrected/superseded; downstream invalidation expected. |
| Withdrawal | Output must no longer be served or cited. | Withdrawn state propagated; public surfaces blocked or marked unavailable. |
| Rollback | A reviewed prior safe release or state is restored in a deterministic drill. | Rollback readiness proof; no production mutation from a unit test. |
| Positive control | A known-safe bounded case proves that a deny/fail rule is not indiscriminately rejecting everything. | Expected pass or allowed bounded outcome. |
| Negative control | A known-unsafe case proves the gate actually blocks the targeted failure mode. | Expected fail/deny/abstain/quarantine/error. |

### Minimum cross-cutting cases

- aggregate source presented as field truth → deny or fail;
- modeled/derived vegetation index presented as direct observation → fail or abstain;
- Soil moisture product role/depth/time collapsed → fail or abstain;
- SSURGO/gSSURGO lineage or vintage mismatch → fail/quarantine;
- missing `EvidenceBundle` for claim-bearing output → abstain;
- rights or sensitivity unknown → hold/deny;
- unpublished or superseded candidate requested by public surface → deny/abstain;
- policy evaluator unavailable → error/fail closed;
- correction upstream without downstream invalidation → fail;
- rollback target absent or non-verifiable → fail/hold;
- denied response leaks protected values → fail even when decision is `DENY`;
- live network attempt in default suite → error/fail.

## Rights, sensitivity, security, and public-safe test data

Agriculture tests can touch high-consequence data categories even when the domain is not generally classified as sensitive. The most restrictive relevant posture wins.

### Deny-by-default test material

Do not commit real or plausibly real:

- living-person or operator identity;
- exact farm or field geometry tied to a private party;
- parcel ownership, title, genealogy, residence, or contact data;

- restricted source payloads or credentials;
- rare-species, rare-plant, archaeology, sacred/cultural, or critical-infrastructure exact locations joined to Agriculture;

Use:

- clearly synthetic identifiers and values;
- generalized or fake geometries;
- aggregate county, watershed, region, or synthetic grid cells when the test does not require exact precision;
- minimized fields;
- canary values that expose leakage;
- explicit aggregation/redaction/generalization expectations;
- safe reason codes and bounded diagnostics;
- deterministic hashes and fixture versions when needed.

### Secret and injection posture

- Never store tokens, passwords, cookies, API keys, private keys, private endpoints, or credentials in tests or fixtures.
- Treat source text, PDFs, HTML, CSV, comments, and payload fields as untrusted data, not instructions to the test runner or AI authoring tool.
- Do not execute repository content merely because a fixture or document asks for it.
- Keep network and subprocess access deny-by-default in fixture-driven tests.
