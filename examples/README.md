## Purpose

`examples/` is the KFM responsibility root for worked demonstrations, walkthroughs, review aids, and deliberately bounded example assemblies.

A useful example should show at least one of the following:

- the smallest understandable success path;
- a negative path such as `ABSTAIN`, `DENY`, `ERROR`, `HOLD`, or `QUARANTINE`;
- the distinction between illustrative shape and operational authority;
- the relationship between an example and the correct contract, schema, policy, fixture, test, app, package, data, proof, receipt, or release home;
- how source role, evidence, rights, sensitivity, review, release, correction, and rollback remain visible where material.

Examples make doctrine inspectable. They become harmful when readers cannot tell where illustration ends and governed behavior begins.

## What belongs here

Accepted material is small, inspectable, deterministic where practical, synthetic or safely redacted, and visibly marked as an example.

| Material | Required posture |
|---|---|
| Worked walkthroughs | State scenario, assumptions, expected outcome, operational home, and non-goals. |
| Minimal payload sketches | Include visible example markers and synthetic identifiers. |
| Negative-state examples | Use finite outcomes and reason codes without leaking restricted detail. |
| Example configuration or code | Include an entrypoint, dependency boundary, expected output, cleanup, and failure behavior before being called runnable. |
| Example EvidenceRef/EvidenceBundle relationships | Illustrative only; proof support remains under `data/proofs/`. |
| Source-intake and receipt sketches | Illustrative only; source and process authority remain in registry, connectors, RAW/QUARANTINE, and receipts. |
| UI, story, Focus, map, or style examples | Show governed inputs and trust-visible states; never normalize internal-store reads or style-as-policy. |
| Domain examples | Preserve owning-domain source roles and sensitivity boundaries. |
| Local README files | Define scope, accepted material, exclusions, validation, correction, retirement, and evidence limits. |

Recommended structured marker:

```json
{
  "example": true,
  "authority": "non_authoritative_example",
  "do_not_publish": true,
  "example_id": "kfm://example/NEEDS-VERIFICATION",
  "expected_outcome": "ABSTAIN",
  "reason": "illustrative example; operational evidence and behavior remain unverified"
}
```

## What does NOT belong here

| Do not place here | Correct home or action |
|---|---|
| RAW source payloads, downloads, API dumps, scans, rasters, vectors, or source mirrors | `data/raw/`, `data/work/`, or `data/quarantine/` according to state |
| Processed data, catalog records, triplets, graph exports, release candidates | The corresponding `data/` phase or `release/candidates/` |
| EvidenceBundles, ProofPacks, validation reports, citation-validation records | `data/proofs/` |
| Run, ingest, transform, validation, AI, telemetry, correction, rollback, or render receipts | `data/receipts/` |
| Published PMTiles, GeoParquet, GeoJSON, COGs, API payloads, stories, reports, or downloads | `data/published/` after release gates close |
| ReleaseManifest, PromotionDecision, CorrectionNotice, WithdrawalNotice, RollbackCard, signatures, release changelog | `release/` |
| SourceDescriptors, source activation decisions, source registry records | `data/registry/`, `control_plane/`, or the accepted registry home |
| Contracts, schemas, policy, validators, tests, fixtures, apps, packages, pipelines, connectors, workflows | Their canonical responsibility roots |
| Secrets, credentials, private tokens, proprietary terms, restricted payloads | Never commit; quarantine or follow the incident/secret-handling path |
| Exact sensitive locations, living-person data, DNA/genomic records, archaeology or sacred-site clues, rare-species locations, critical-infrastructure detail, private parcel joins, reconstructive redaction clues | Synthesize, generalize, redact, restrict, delay, quarantine, or deny |

## Inputs

Examples may be authored from:

- accepted doctrine, ADRs, contracts, schemas, policy, and source standards;
- current implementation and bounded tests when the example is explicitly documenting real behavior;
- synthetic public-safe data designed for a stated learning objective;
- safely redacted or generalized material with transform rationale;
- verified fixtures copied or linked under an explicit example/fixture relationship;
- review findings, failure reports, or correction cases rewritten as non-sensitive teaching scenarios.

Before admitting an input, record:

1. its source and authority class;
2. whether values are real, synthetic, transformed, or placeholders;
3. rights and redistribution posture;
4. sensitivity and harmful-precision posture;
5. the operational object or behavior the example must not replace;
6. the expected finite outcome and known limitations.

Unknown rights, unclear source role, unresolved sensitivity, or unverifiable provenance blocks using real material. Prefer a synthetic substitute.

## Outputs

`examples/` may emit documentation-facing artifacts such as:

- a worked Markdown walkthrough;
- a small synthetic JSON/YAML/configuration sketch;
- a runnable local demonstration package after the [runnable admission contract](#runnable-example-admission-contract) closes;
- expected example output and cleanup instructions;
- a review checklist or failure-state matrix;
- a candidate proposed for separate fixture/test graduation.

It must not emit or claim:

- canonical data or source records;
- operational receipts or proofs;
- policy or release decisions;
- production runtime responses;
- deployed services or public endpoints;
- KFM `PUBLISHED` state.

### Promotion out of examples

When an example becomes operationally important, create or move the operational artifact into its owning root through a reviewed change. Keep a minimal example only when it still teaches something and its lineage/update burden is explicit.

## Example maturity model

The following vocabulary is **PROPOSED** for consistent reporting; it is not a claim that a lane has graduated.

| State | Minimum evidence | Must not be inferred from |
|---|---|---|
| `README_ONLY` | Boundary README exists. | Directory name alone. |
| `STATIC_WALKTHROUGH` | Reviewable synthetic document/payload with expected outcome and non-authority markers. | Polished prose alone. |
| `STRUCTURE_VALIDATED` | Accepted schema/contract reference and observed validation result. | Parseability alone. |
| `RUNNABLE_LOCAL` | Reproducible command, pinned dependencies/runtime, deterministic inputs/outputs, cleanup, negative case, observed run. | A code block or filename. |
| `FIXTURE_MIRRORED` | Separate fixture/test artifact with lineage, drift check, and consumer agreement test. | Copying an example into `fixtures/`. |
| `STALE` | Referenced behavior or contract changed and the example no longer agrees. | Age alone. |
| `RETIRED` | Consumer/reference review completed; replacement or historical note recorded; deletion/redirect reviewed. | Silent abandonment. |

A runnable example must declare:

```text
example identity and purpose
supported runtime and exact entrypoint
pinned or locked dependencies
synthetic/no-network default inputs
expected deterministic output
negative/failure example
cleanup and side effects
security, rights, and sensitivity posture
owning implementation and contract references
smoke-test command and observed result
correction and retirement path
```

It must not write canonical data, mutate lifecycle stores, contact a live source, call a model provider, publish artifacts, or require secrets by default unless a separately governed profile explicitly permits and audits that behavior.

## Example contract

Every consequential example should answer:

| Question | Expected answer |
|---|---|
| What scenario is illustrated? | A bounded synthetic or safely transformed scenario. |
| What is real versus synthetic? | Explicit field- or section-level declaration. |
| What operational object/behavior is represented? | A precise reference to the owning root. |
| What evidence and source role apply? | Synthetic refs or verified support; roles do not collapse. |
| What policy/sensitivity state applies? | A finite illustrative state and safe failure behavior. |
| What outcome should render? | `ANSWER`, `ABSTAIN`, `DENY`, `ERROR`, `HOLD`, `QUARANTINE`, or a documented non-runtime state. |
| What is the release posture? | Normally `not_released`; an operational release ref only when verified outside examples. |
| How is it validated? | Exact bounded checks matching the claimed maturity. |
| How is it corrected or retired? | Owner, trigger, reference search, replacement/rollback plan. |
| What must not happen? | No operational authority or publication by example placement. |

### Finite-outcome rules

- Missing, stale, conflicting, or citation-invalid evidence → `ABSTAIN`.
- Rights, sensitivity, role, or public-safety prohibition → `DENY` or `HOLD` as the applicable contract defines.
- Tool/schema/adapter/runtime failure → `ERROR`.
- Quarantined source material → `QUARANTINE`/`HOLD`, never a weaker public answer.
- `ANSWER` requires evidence, policy, review, release, and citation posture appropriate to the scenario.

## Sensitive data and safety posture

Examples are public-review material by default, so the safe baseline is stricter than “not production.”

| Risk | Required example posture |
|---|---|
| Living-person or DNA/genomic material | Synthetic only unless explicit governed authorization proves a safer path. |
| Archaeology, burial, sacred/cultural places | No exact locations or reconstructive clues; generalize, redact, restrict, or deny. |
| Rare species or sensitive ecology | No exact occurrences or join patterns that recover them. |
| Critical infrastructure | No vulnerability, dependency, condition, operator-sensitive, or restricted geometry detail. |
| Private parcels/land-title joins | Synthetic/generalized; no person-parcel exposure. |
| Source terms/credentials | Never include credentials; verify redistribution/attribution or synthesize. |
| Model prompts or outputs | No hidden reasoning, private prompt logs, provider transcripts, or raw model output as evidence. |
| Styling/geoprivacy | Transform sensitive geometry before rendering; style filters and opacity are not protection. |

A redacted example should explain the transform category and why it is safe without publishing the hidden value or a reversible reconstruction recipe.

## Maintenance, correction, and rollback

### Change discipline

1. Identify the learning objective and consumer.
2. Re-read the owning operational contract/implementation.
3. Preserve example identity and stable links when possible.
4. Update positive and negative states together.
5. Re-run the checks matching the claimed maturity.
6. Record what became stale, corrected, superseded, or retired.
7. Search inbound references before moving or deleting.

### Stale example response

| Condition | Action |
|---|---|
| Contract/schema changes | Correct the example or mark `STALE`; do not silently preserve an invalid shape. |
| Runtime behavior changes | Re-run and update expected output before retaining `RUNNABLE_LOCAL`. |
| Rights/sensitivity changes | Remove unsafe content immediately; quarantine or use a synthetic replacement. |
| Example graduates to implementation/fixture | Put the operational artifact in its owning root; retain only a maintained teaching copy or retire it. |
| No consumer or learning value remains | Retire through reviewed deletion with link search and rollback note. |
