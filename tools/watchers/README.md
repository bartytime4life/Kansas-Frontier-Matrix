## Purpose

`tools/watchers/` is the parent routing surface for **reusable watcher tooling**, not the universal home for every file with “watch” in its name.

The durable question for this directory is:

> Is this component a reusable, side-effect-limited observer/checker invoked by governed workflows, or is it actually executable orchestration, declarative configuration, upstream source access, source authority, lifecycle data, policy, evidence, release state, or public runtime behavior?

A watcher may help determine that a source-facing signal changed. It does not establish:

- that the source is admitted or activated;
- that a payload changed materially;
- that domain truth changed;
- that source role, rights, consent, cultural authority, or sensitivity remain valid;
- that evidence is complete;
- that a lifecycle transition is allowed;
- that a release or public notification should occur.

This parent README owns routing, child-lane admission rules, shared non-publisher invariants, and compatibility/migration guidance. It does not own implementation merely by naming the directory.

## Responsibility and placement

Directory Rules assign one primary responsibility:

```text
tools/          repo-wide validator, generator, builder, checker, reusable trust helper
pipelines/      executable pipeline logic and lifecycle orchestration
pipeline_specs/ declarative pipeline configuration
connectors/     source-specific fetcher or admitter
data/registry/  source identity, role, rights, sensitivity, cadence, activation
contracts/      object meaning
schemas/        machine-checkable shape
policy/         allow / deny / restrict / abstain decisions
tests/          enforceability proof
fixtures/       deterministic test examples
data/           lifecycle records, receipts, proofs, catalogs, published artifacts
release/        promotion, correction, withdrawal, supersession, rollback
apps/           governed serving surfaces
```

### Placement test

| Component answers… | Owning home |
|---|---|
| “How do I compare two already-authorized metadata snapshots?” | `tools/watchers/` **may** fit if reusable and side-effect-limited. |
| “How does a watcher run, sequence steps, route candidates, and invoke downstream stages?” | `pipelines/` |
| “What source, cadence, checks, thresholds, and handoffs should a watcher use?” | `pipeline_specs/` |
| “How do I access or download the upstream source?” | `connectors/` |
| “Is this source admitted, in what role, under what rights and sensitivity?” | `data/registry/` plus accepted governance controls |
| “What does the watcher output mean?” | `contracts/` |
| “What fields and enums must the output have?” | `schemas/` |
| “May this run or output proceed?” | `policy/` |
| “Does the behavior fail closed?” | `tests/` and `fixtures/` |
| “Where is the emitted candidate, receipt, or proof?” | accepted `data/` lifecycle/trust-artifact home |
| “May this reach the public?” | `release/` and governed applications |

### Key rule

```text
reusable watcher helper != watcher orchestration
watcher orchestration    != watcher specification
watcher specification    != source activation
source activation        != public release
```

When a component spans multiple responsibilities, split it rather than selecting a convenience directory.

## Watcher component taxonomy

A governed watcher system is a composition of separate components.

| Component | Responsibility | Authority it does not have |
|---|---|---|
| `SourceDescriptor` | Source identity, role, rights, cadence, sensitivity, access posture | Publication or claim truth |
| `SourceActivationDecision` | Whether and how a source may be used | Release approval |
| Watch profile/spec | Declarative source scope, signals, cadence, thresholds, outputs | Execution or activation |
| Connector | Authorized upstream access and bounded capture | Domain truth or publication |
| Watcher helper | Reusable comparison, hashing, freshness, classification, report-building logic | Orchestration, policy, lifecycle promotion |
| Watcher pipeline | Executes steps and routes finite outcomes | Source authority, evidence closure, release |
| Candidate/report | Records observed signal and review need | ValidationReport, EvidenceBundle, catalog truth |
| Run receipt | Auditable process memory | Proof that content is true or publishable |
| Policy decision | Stage-specific allow/restrict/deny/abstain | Evidence or release by itself |
| Release decision | Governs publication/correction/rollback | Source or domain truth by itself |
| Public notification | Released, minimized communication | Direct view of watcher state |

### One implementation rule

For each watcher concern, there must be one active implementation owner. Compatibility lanes may forward, warn, or document migration, but they must not contain independent evolving copies.

## Watcher invariants and anti-collapse

### Core invariants

- Watchers detect possible change; they do not declare truth.
- Watchers are downstream from source admission and upstream from substantive validation and release.
- Source roles are read from governed source records and are never upgraded by watcher output.
- Source-head and metadata signals are weak intake/process evidence.
- Missing rights, consent, cultural authority, sensitivity, baseline, or review support fails closed.
- Joined sensitivity is recomputed for the watcher output and downstream candidate.
- Watcher outputs are finite, reason-coded, reviewable, and reversible.
- Public clients never consume watcher state directly.
- AI may summarize an already-governed watcher result but cannot create evidence or approval.

### Disallowed collapses

```text
path exists                       -> active watcher
README exists                     -> implementation proof
spec merged                       -> scheduled execution
schedule exists                   -> source activation
watcher invoked                   -> source fetch authorized
HTTP 200                          -> source admitted
ETag changed                      -> payload materially changed
payload changed                   -> domain truth changed
checksum matched                  -> rights or sensitivity approved
candidate report                  -> ValidationReport
watcher receipt                   -> EvidenceBundle
review handoff                    -> release candidate
watcher success                   -> catalog update
watcher notification              -> official public alert
generated summary                 -> evidence
```

### Self-approval prohibition

A watcher, its helper, its pipeline, and its generated output must not approve their own:

- source activation;
- policy decision;
- evidence closure;
- release;
- correction;
- rollback completion.

## Source and run admission

A live watcher run requires explicit, resolvable prerequisites.

| Gate | Required support | Fail-closed posture |
|---|---|---|
| Identity | Stable watcher id, version, owner, domain/source scope | `WATCHER_IDENTITY_MISSING` |
| Source descriptor | Source id, role, authority, rights, sensitivity, cadence, access, steward | `WATCHER_SOURCE_DESCRIPTOR_MISSING` |
| Activation | Current allowed/restricted/denied/review state with scope | `WATCHER_SOURCE_ACTIVATION_MISSING` |
| Specification | Immutable profile ref/digest and accepted parser | `WATCHER_SPEC_MISSING` |
| Executable owner | One verified helper/pipeline implementation | `WATCHER_IMPLEMENTATION_UNRESOLVED` |
| Connector | One accepted source-access path when live access is required | `WATCHER_CONNECTOR_UNRESOLVED` |
| Baseline | Prior state, manifest, source-head, snapshot, or explicit no-baseline rule | `WATCHER_BASELINE_MISSING` |
| Rights and consent | Current terms, attribution, redistribution, consent obligations | `WATCHER_RIGHTS_UNRESOLVED` |
| Sensitivity and sovereignty | Sensitive classes, cultural authority, geoprivacy, private-data controls | `WATCHER_SENSITIVITY_UNRESOLVED` |
| Fixture parity | Deterministic no-network cases and expected outputs | `WATCHER_FIXTURE_MISSING` |
| Review path | Named source/domain/security/sensitivity reviewer and queue | `WATCHER_REVIEW_PATH_MISSING` |
| Output route | Accepted candidate/report/receipt destinations | `WATCHER_OUTPUT_ROUTE_UNRESOLVED` |
| Rollback | Kill switch, prior version, correction and rollback target | `WATCHER_ROLLBACK_MISSING` |

The proposed source-admission ADR describes descriptor and activation machinery, but its status is `proposed`. Implementations must verify the accepted current vocabulary and storage paths rather than copying proposed values uncritically.

## Network, scheduling, and side effects

### Default posture

```text
development = no network
tests       = no network
CI          = no network unless explicitly approved
live watch  = denied until all admission and security gates close
```

### Live-network prerequisites

A live watcher must define:

- allowed hosts, protocols, methods, paths, and redirect behavior;
- authentication source without storing credentials in repository files;
- timeout, retry, backoff, jitter, rate-limit, and circuit-breaker behavior;
- metadata-first observation before large or sensitive downloads;
- maximum response size and content-type validation;
- temporary storage and cleanup;
- logs and telemetry minimization;
- kill switch and incident containment;
- fixture-equivalent replay;
- no-publication and no-promotion assertions.

### Scheduler rules

A schedule must not:

- imply activation;
- run against a draft or denied source;
- widen source scope;
- bypass no-network test parity;
- perform automatic merge, approval, or release;
- silently retry indefinitely;
- hide skipped or stale runs;
- treat cadence completion as freshness proof.

### Side-effect budget

Reusable watcher helpers under `tools/` should be pure or explicitly bounded by default. Any filesystem, network, subprocess, queue, issue, pull-request, notification, or lifecycle write must be declared and tested.

### Input rules

- No ambient source activation inferred from file presence.
- No hidden baseline discovered by scanning arbitrary directories.
- No source role inferred from a URL, filename, popularity, or AI summary.
- No credentials embedded in the input packet.
- Sensitive input references use opaque identifiers where practical.
- Every mutable input has a version, digest, or transaction-time reference.

## Bounded output contract

### Allowed output families

A watcher may emit or prepare:

- `NoChange` / no-op result;
- source-head or metadata observation;
- bounded material-change candidate;
- source-intake or proposed-work candidate;
- quarantine or hold handoff;
- steward review summary;
- reason-coded restriction, denial, abstention, or error;
- run receipt and immutable references.

### Required properties

Every output should preserve:

- watcher, profile, implementation, and source identity;
- input refs and hashes;
- source role without upgrade;
- observed, retrieval, run, review, and release time distinctions where material;
- signal classes checked;
- changed and unchanged findings;
- materiality classification and reason codes;
- rights, consent, cultural-authority, sensitivity, and security flags;
- candidate/non-public state;
- review requirements;
- correction, supersession, and rollback references where applicable.

### Forbidden output behavior

A watcher must not directly emit or mutate:

- source activation records;
- admitted RAW payloads of record unless a separately governed connector owns that write;
- normalized or processed domain records;
- ValidationReports masquerading as watcher results;
- EvidenceBundles or proof closure;
- canonical catalog or triplet records;
- release manifests or promotion decisions;
- public layers, APIs, tiles, exports, graphs, search indexes, Focus Modes, or AI answers;
- automatic approval, merge, or publication.

|---|---|
| `WATCHER_NO_CHANGE` | Allowed signals were checked and no material candidate was detected. |
| `WATCHER_CHANGE_CANDIDATE` | A bounded change requires downstream review. |
| `WATCHER_NEEDS_REVIEW` | Human or steward interpretation is required. |
| `WATCHER_HOLD` | A prerequisite, baseline, policy, or review dependency is incomplete. |
| `WATCHER_QUARANTINE` | The candidate is isolated because risk or validity cannot be resolved. |
| `WATCHER_RESTRICT` | The run or output is allowed only under explicit constraints. |

| `WATCHER_DENY` | The requested run, write, join, notification, or exposure is prohibited. |


Domain-specific reason codes may extend this vocabulary but must not silently redefine shared meanings.

## Rights, sensitivity, sovereignty, and security

A watcher output is a new derived product. Its rights and sensitivity are not automatically equal to the least restrictive input.

### High-risk classes

Use deny, hold, quarantine, redaction, generalization, staged access, or private review for watcher signals involving:

- exact rare-species or rare-plant locations;
- archaeology, burial, sacred, cultural, or Indigenous knowledge;
- living-person, genealogy, identity, consent, or DNA/genomic data;
- private land, stewardship, or source-restricted field notes;
- critical infrastructure or operationally sensitive transport/facility information;
- credentials, private endpoints, signing material, or security defects;
- small-cell, differencing, triangulation, or reconstruction risk;
- proprietary, licensed, embargoed, or terms-limited sources.

### Log and notification minimization

Watcher logs and notifications should contain only:

- opaque watcher/run/source identifiers;
- high-level reason codes;
- minimized counts or classes;
- review route;
- non-sensitive timestamps;
- artifact refs that enforce access.

They should not contain exact coordinates, restricted names, private identifiers, raw excerpts, secret-bearing URLs, geoprivacy parameters, or reconstruction hints.

### Security reporting

A watcher or workflow that could expose sensitive information, publish directly, leak credentials, or bypass lifecycle/policy/release gates is a security-sensitive issue. Follow `SECURITY.md`: report privately, preserve minimal evidence, and contain the affected watcher or schedule.

## Lifecycle and non-publication boundary

Watcher activity precedes governed lifecycle processing.

```text
admitted source + accepted profile
  -> watcher signal comparison
  -> no-op / candidate / hold / quarantine / deny / abstain / error
  -> connector or ingest decision when source capture is required
  -> RAW
  -> WORK / QUARANTINE
  -> PROCESSED
  -> CATALOG / TRIPLET
  -> release decision
  -> PUBLISHED public-safe artifact
```

A watcher must not collapse this into:

```text
source changed -> update public layer
```

### Writer boundary

| Destination | Watcher posture |
|---|---|
| Temporary fixture/test directory | Allowed when deterministic and cleaned up. |
| Accepted candidate/report lane | Allowed when explicitly configured and reviewed. |
| Accepted receipt lane | Allowed through a defined receipt emitter. |
| RAW | Only through separately governed connector/admission ownership. |
| WORK / QUARANTINE | Handoff may be prepared; lifecycle writer must be explicit. |
| PROCESSED | Denied. |
| CATALOG / TRIPLET | Denied. |
| `release/` | Denied. |
| PUBLISHED / public applications | Denied. |

## Validation, tests, fixtures, and CI

### Required test families

| Test family | Required proof |
|---|---|
| Placement | Helper, pipeline, profile, connector, registry, policy, and output files remain in owning roots. |
| Identity | Watcher/profile/source ids and versions are stable and explicit. |
| Activation | Missing, denied, restricted, retired, and stale activation states fail correctly. |
| No-network | Default unit and CI tests cannot access live hosts. |
| Determinism | Same inputs and clock produce same output and hashes. |
| Baseline | Missing, incompatible, corrupt, and superseded baselines produce finite outcomes. |
| Materiality | Changed metadata does not automatically become material domain change. |
| Source role | Watcher output cannot upgrade or invent source role. |
| Rights and consent | Unknown or changed terms block unsafe use. |
| Sensitivity | Exact/reconstructable sensitive details are absent from outputs and logs. |
| Security | Credentials, private endpoints, and secret-bearing errors are redacted. |
| Side effects | Undeclared network, filesystem, subprocess, issue, PR, or notification writes fail. |
| Output path | Direct writes to processed/catalog/triplet/release/published/public paths fail. |
| Review | Required reviewer routes and separation of duties are enforced. |
| Correction | Corrected or superseded watcher results preserve lineage. |
| Rollback | Schedule and implementation can be disabled and prior state restored. |

### Fixture rules

Fixtures must be:

- synthetic or appropriately minimized;
- deterministic;
- no-network;
- explicit about expected outcome and reason code;
- free of real credentials, private records, and exact sensitive locations;
- paired with valid and invalid cases;
- versioned when used as regression anchors.

### CI admission

A watcher is not implemented because:

- a README exists;
- a profile parses;
- a schedule is present;
- a fixture example exists;
- a TODO workflow is green.

Substantive CI must execute the accepted parser and implementation against positive and negative fixtures, enforce no-network behavior, verify output destinations, and fail closed on sensitive/publication bypasses.

## Review, operations, and incident posture

### Review burden

Watcher changes require reviewers appropriate to impact:

- tooling/watcher steward;
- source and connector steward;
- affected domain steward;
- rights/consent reviewer;
- sensitivity/geoprivacy reviewer;
- cultural/sovereignty reviewer when applicable;
- security reviewer for network, credentials, logging, or exposure;
- pipeline/spec steward;
- release steward when downstream artifacts may be affected.

Current CODEOWNERS is a placeholder and does not enforce this review burden.

### Operational record

Each implemented run should make inspectable:

- watcher/profile/implementation/source versions;
- activation and policy refs;
- baseline ref;
- signals checked;
- outcome and reason codes;
- input/output hashes;
- side effects;
- reviewer handoff;
- correction/supersession refs;
- rollback target.

### Incident response

When a watcher behaves unsafely:

1. disable the schedule and live-network path;
2. block downstream consumers;
3. preserve minimized logs, inputs, outputs, hashes, and decisions;
4. quarantine affected candidates;
5. rotate exposed credentials;
6. audit public and internal exposure;
7. correct the source/profile/code/policy/fixture through review;
8. replay deterministically;
9. issue correction or withdrawal records for affected downstream products;
10. reactivate only after fail-closed tests and review pass.
