## Purpose

`pipeline_specs/settlement/` is the compatibility alias for declarative Settlement-sublane run intent.

Its safe role is narrow:

- explain why the shorter alias exists;
- bind any future alias profile to the governing `settlements-infrastructure` lane;
- document which legal-status, operational-status, source-role, temporal, evidence, sensitivity, representation, review, receipt, release, correction, and rollback gates a future alias profile would have to satisfy;
- prevent the alias from becoming a second source of pipeline-spec authority.

This directory does **not** currently establish an active Settlement specification. The bounded inventory found only this README.

### Audience

- pipeline-spec and Settlements/Infrastructure maintainers;
- settlement identity, municipal/legal-status, infrastructure, service-area, operator, condition, dependency, historic-community, and source-vintage stewards;
- source, rights, sensitivity, temporal, evidence, policy, validation, release, security, and documentation reviewers;
- maintainers implementing a future schema, parser, registry, loader, scheduler, or executable consumer;
- reviewers preventing the `settlement` alias from becoming a parallel authority or a shortcut around infrastructure sensitivity controls.

## Authority and anti-collapse

### Responsibility split

```text
pipeline_specs/  = declarative run intent: WHAT may run and under which gates
pipelines/       = executable behavior: HOW processing occurs
connectors/      = source fetch and admission support; never publication
configs/         = safe-to-commit consumer settings; never secrets or activation authority
data/            = lifecycle state, registries, receipts, proofs, catalog/triplets, and published artifacts
contracts/       = semantic meaning
schemas/         = machine-checkable shape
policy/          = admissibility, rights, sensitivity, access, and release obligations
tests/fixtures/  = enforceability proof and controlled synthetic/generalized examples
release/         = release, correction, supersession, withdrawal, takedown, and rollback authority
apps/            = governed serving surfaces; never direct reads from specs or internal stores
```

Directory Rules assign declarative configuration to `pipeline_specs/` and executable pipeline behavior to `pipelines/`. `settlements-infrastructure` is a domain segment inside responsibility roots, not a root.

### What this README may decide

This README may define the maintenance boundary for the alias:

- what belongs here;
- what must remain under another responsibility root;
- what repository evidence is currently verified;
- what a future alias profile would have to contain;
- which legal-status, operational-status, temporal, sensitivity, evidence, and release distinctions must survive;
- what remains `UNKNOWN`, `CONFLICTED`, or `NEEDS VERIFICATION`;
- how a documentation-only revision is validated and rolled back.

### What this README cannot decide

This README cannot:

- admit, activate, suspend, retire, or supersede a source;
- determine incorporation, annexation, dissolution, jurisdiction, ownership, service obligation, facility operation, condition, dependency, or emergency status;
- define a canonical `Settlement`, `Municipality`, `Facility`, or other object;
- satisfy rights, sensitivity, evidence, policy, review, or release gates merely by naming them;
- execute a pipeline or move data between lifecycle states;
- create an `EvidenceBundle`, policy decision, release manifest, correction notice, or rollback card;
- authorize public facility detail, private-property joins, critical dependency exposure, or exact sensitive locations;
- approve its own alias status or migration.

### Forbidden collapses

```text
README or spec path              -> active configuration
valid YAML                        -> admitted or executable profile
source list                       -> source authority
GNIS or gazetteer record          -> current legal settlement status
CensusPlace                       -> Municipality
SettlementCandidate               -> incorporated Settlement
historic townsite                 -> present occupied community
mapped structure                  -> active Facility
ConditionObservation              -> current evergreen operational status
ServiceArea                       -> guaranteed service availability
Dependency edge                   -> public-safe exposure surface
operator record                   -> release-approved operator detail
generalized geometry              -> exact canonical geometry
catalog or triplet projection     -> sovereign truth
workflow success                  -> evidence, policy, or release approval
```

## Bounded context and ubiquitous language

The governing bounded context is **Settlements / Infrastructure**, represented by the repository segment `settlements-infrastructure`.

`settlement` is a shorter compatibility alias. It does not create a separate model.

| Term | Meaning in this README | Boundary |

## Repository fit and alias discipline

### Working authority relationship

```text
pipeline_specs/settlements-infrastructure/   # governing declarative lane
pipeline_specs/settlement/                   # compatibility alias; README-only at snapshot

pipelines/domains/settlements-infrastructure/ # governing executable lane
pipelines/domains/settlement/                 # compatibility executable alias boundary
```

### Alias invariants

1. The alias must bind to one governing `settlements-infrastructure` specification identity.
2. The alias must not fork source lists, legal-status rules, sensitivity rules, object semantics, lifecycle paths, receipts, release rules, or public surfaces.
3. Alias resolution must be deterministic and auditable.
4. A missing, ambiguous, cyclic, stale, or version-incompatible canonical target must produce a fail-closed result.
5. Alias state must remain visible in validation reports and receipts.
6. An alias cannot activate a canonical profile.
7. Migration requires an ADR or accepted equivalent decision, explicit path map, compatibility window, tests, documentation, deprecation state, and rollback.
8. The current Directory Rules copy conflict is surfaced, not resolved by this README.

### Why the alias remains README-only

The repository evidence does not establish a unique need for executable or machine-consumed alias artifacts. Adding alias YAML now would increase:

- duplicate-authority risk;
- parser and scheduler ambiguity;
- source and legal-status drift;
- sensitivity-policy divergence;
- lifecycle and receipt fragmentation;
- migration and rollback burden.

The smallest safe posture is documentation-only until the need is proven.

## State and activation model

No canonical state vocabulary or activation mechanism was established. A safe future model should distinguish at least:

```text
DRAFT
  -> REVIEW
  -> VALIDATED
  -> APPROVED_FOR_ACTIVATION
  -> ACTIVE
  -> SUSPENDED
  -> SUPERSEDED
  -> RETIRED
```

### State is not activation

These events do not independently activate a spec:

- committing or merging a file;
- passing YAML or schema validation;
- resolving an alias;
- listing sources;
- creating a schedule;
- completing a workflow;
- producing a candidate receipt;
- setting `status: active` inside untrusted content.

Activation should require a separate, auditable decision that binds:

- exact spec digest/version;
- canonical target and executable consumer;
- admitted sources;
- rights and sensitivity decisions;
- legal/operational-status policy;
- validated fixtures and negative cases;
- reviewer roles;
- effective time and expiry;
- deactivation and rollback target.

### Fail-closed activation outcomes

| Condition | Required outcome |
|---|---|
| Alias target missing, ambiguous, cyclic, or incompatible | `ERROR` or `HOLD` |
| Source descriptor missing or unusable for the claim type | `DENY`, `ABSTAIN`, or `QUARANTINE` |
| Legal-status authority unresolved | `ABSTAIN` or restricted candidate only |
| Operational status stale or unsupported | `ABSTAIN`, `STALE`, or `QUARANTINE` |
| Rights or sensitivity unresolved | `DENY` or `HOLD` |
| Critical infrastructure/dependency exposure unresolved | `DENY` |
| Evidence closure missing | `ABSTAIN` |
| Release approval missing | no published output |

## Sources, roles, rights, and authority

A future alias profile may reference canonical `SourceDescriptor` identities. It cannot admit a source or create a separate source registry.

### Relevant source families

- Census TIGER and census-place geography;
- GNIS and other gazetteers;
- state and local GIS;
- municipal charters, ordinances, annexation records, dissolution records, and other legal records;
- historical gazetteers, maps, directories, newspapers, and archival records;
- infrastructure operators and providers;
- transportation, bridge, facility, utility, service-area, hazards, and resilience sources;
- field or remote-sensing observations when admissible and appropriately scoped.

Current rights, endpoints, cadence, source terms, public-release posture, and authority scope remain source-specific and **NEEDS VERIFICATION**.

| Source or representation | Permitted role examples | Must not silently become |

| Municipal charter, ordinance, state record | legal/administrative authority for stated scope and valid time | universal current truth outside its authority |
| Census place/TIGER geography | statistical/administrative representation | incorporated municipality |
| GNIS or gazetteer | name/location context | legal status or current occupation |
| Historical map or directory | historical/contextual evidence | present-day status |
| Local GIS feature | geometry/context/administrative representation | title, ownership, operation, or legal incorporation |
| Operator/provider record | operator or service context | unrestricted infrastructure disclosure or service guarantee |
| Condition inspection/observation | time-bounded observation | current evergreen operational status |
| Model or inferred settlement/facility | candidate/model result | observed or authoritative object |
| Map/tile/graph projection | downstream carrier | canonical evidence |

A profile that cannot preserve the source's actual role must reject, quarantine, or abstain.

### Rights and attribution

A future profile should bind:

- allowed access method;
- permitted use and redistribution;
- attribution/citation requirements;
- retention and deletion obligations;
- embargo or delayed-access rules;
- sensitivity and exact-location restrictions;
- source-vintage and refresh expectations;
- downstream derivative constraints.

Unclear rights block higher-risk processing and publication.

## Settlement identity and legal-status boundaries

Settlement identity resolution is not string matching.

A future profile must preserve distinctions among:

- name assertion;
- alternate/historic name;
- settlement candidate;
- settlement identity candidate;
- census place;
- incorporated municipality;
- unincorporated community;
- townsite or plat;
- ghost town;
- fort, mission, or reservation community;
- historical versus current status;
- observed place versus modeled or inferred candidate.

### Legal-status gate

A legal-status claim should require:

1. an appropriate authoritative source role;
2. the legal act or record supporting the status;
3. valid-time and effective-date handling;
4. geography/version context;
5. review state;
6. evidence closure;
7. correction and supersession path.

### Examples of required abstention

- a census place shares a name with a municipality but legal identity is unresolved;
- a GNIS feature exists but current occupation is unknown;
- a historic map shows a townsite but present status is unsupported;
- annexation or dissolution effective dates conflict;
- the feature geometry and legal boundary are from different vintages;
- multiple same-name settlements cannot be deterministically resolved.

### Identity and geometry

A matching point, polygon, parcel, or label does not alone establish identity. Any proposed deterministic identity rule remains **NEEDS VERIFICATION** until the relevant contract, schema, canonicalization, crosswalk, and tests are inspected.

## Infrastructure and operational-status boundaries

The alias must not become an infrastructure disclosure lane.

Infrastructure-sensitive objects include:

- assets, facilities, network nodes, and network segments;
- operators and provider relationships;
- service areas and access zones;
- condition observations;
- dependency and interdependency relations;
- exact geometry, access points, control systems, vulnerability context, outage context, and capacity details.

### Operational-status anti-collapse

```text
feature exists in source     != currently operating
condition observed at T1     != condition at T2
operator named historically  != current operator
service area polygon         != guaranteed service
dependency inferred          != verified dependency
public map point              != unrestricted exact geometry
absence from source           != asset absent
```

### Public-safe representation

Before public exposure, the governing pipeline may need to:

- remove or generalize exact geometry;
- suppress operator-sensitive or security-significant fields;
- aggregate facility classes or service coverage;
- delay stale or incident-sensitive information;
- remove private-property and living-person joins;
- exclude dependency/vulnerability details;
- attach a `RedactionReceipt`, `GeneralizationReceipt`, or accepted equivalent;
- preserve the source and canonical restricted representation separately;
- bind the public derivative to a release manifest and rollback target.

This README cannot decide which transformation is sufficient for a given asset class.

## Temporal and source-vintage semantics

Settlement and infrastructure claims are time-aware.

A future profile should keep distinct, where material:

| Time kind | Meaning |
|---|---|
| Observation time | When a feature, condition, event, or state was observed |
| Valid time | When the asserted status or geometry applies |
| Legal effective time | When incorporation, annexation, dissolution, jurisdiction, or other legal status takes effect |
| Source publication/vintage | Which source edition or snapshot supplied the representation |
| Retrieval time | When KFM acquired the source |
| Processing time | When KFM transformed or validated it |
| Review time | When a steward reviewed the claim or derivative |
| Release time | When an artifact became governed/public |
| Correction/supersession time | When prior content was corrected, withdrawn, or superseded |

### Stale-state requirements

A profile should define:

- freshness tolerance per source and claim type;
- behavior when source vintage is missing;
- whether stale material may remain as historical context;
- whether stale operational context produces `ABSTAIN`, `STALE`, `QUARANTINE`, or `DENY`;
- how new legal records supersede prior status;
- how geometry changes propagate to catalogs, graph projections, maps, caches, exports, and AI answers.

A schedule is not freshness proof. A recent retrieval does not make an old source current.

## Lifecycle gates and finite outcomes

The alias may describe intent but cannot perform lifecycle transitions.

```text
RAW -> WORK / QUARANTINE -> PROCESSED -> CATALOG / TRIPLET -> PUBLISHED
```

### Gate expectations

| Gate | Required question |
|---|---|
| Source | Is each source admitted for this use, role, rights, sensitivity, and time scope? |
| Alias | Does the profile resolve deterministically to one canonical target? |
| Shape | Does the input conform to the accepted schema? |
| Meaning | Are settlement, legal-status, facility, condition, service-area, and dependency distinctions preserved? |
| Temporal | Are valid time, legal effective time, observation time, source vintage, and stale behavior explicit? |
| Evidence | Can consequential claims resolve to an admissible `EvidenceBundle`? |

| Representation | Is the intended output restricted, generalized, redacted, aggregate, or public-safe as required? |

| Outcome | Meaning |
|---|---|

| `ERROR` | Deterministic operational failure; do not fall back to allow. |

## Watchers, dry runs, and no-op discipline

A watcher or scheduled profile is not present in this alias lane.

Any future watcher must remain a non-publisher.

### Watcher may

- compare source metadata and content hashes;
- identify changed legal records, geometries, names, conditions, service areas, or operator records;
- emit a material-change candidate;
- route ambiguous or sensitive changes to review;
- produce a deterministic `NO_OP` when nothing material changed;
- open a reviewable candidate or pull request when explicitly authorized.

### Watcher must not

- create legal status from source presence;
- infer current operation from an update;
- expose new facility/dependency geometry automatically;
- change source authority or sensitivity tier;
- promote data between lifecycle stages without governed transitions;
- publish, update current aliases, or write directly to public artifacts;
- treat an upstream deletion as verified real-world deletion without review;
- use network credentials embedded in the spec.

### Dry-run minimums

A dry run should show:

- exact spec/canonical target and input digests;
- source versions and retrieval metadata;
- candidate changes and unchanged counts;
- legal-status and temporal conflicts;
- sensitivity and representation decisions;
- expected receipts and output paths;
- reasons for `DENY`, `ABSTAIN`, `QUARANTINE`, `HOLD`, `NO_OP`, or `ERROR`;
- no mutation of canonical or published state.

## Security, logging, network, and representation

### Never commit or log

- credentials, API keys, tokens, cookies, or private endpoints;
- restricted facility access details or vulnerability topology;
- exact sensitive asset, burial, sacred, cultural, private-property, or living-person coordinates;
- private operator records or contractual details;
- restricted source payloads;
- unreleased RAW, WORK, or QUARANTINE data;
- unredacted dependency graphs;
- secrets embedded in example environment variables or URLs.

### Network posture

Default CI should use deterministic no-network fixtures. Live-network execution should require:

- explicit activation;
- reviewed source and endpoint identity;
- least-privilege credentials from approved secret management;
- timeouts, retries, rate limits, and bounded pagination;
- source terms and attribution;
- immutable or reproducible capture where permitted;
- quarantine on malformed, ambiguous, restricted, or unsupported responses;
- redacted logs and receipts.

### Map and representation posture

A map feature is a carrier, not truth. Public map, Evidence Drawer, Focus Mode, and export surfaces should receive released, policy-filtered, public-safe representations through governed interfaces. The alias spec must never supply direct browser pointers to canonical/internal stores.

## Correction, deactivation, and rollback

A pipeline spec must be correctable and reversible.

- source authority or rights change;
- legal status corrected or superseded;
- municipality/census-place identity conflict resolved;
- source-vintage or geometry mismatch;
