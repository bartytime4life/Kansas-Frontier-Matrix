# Kansas Frontier Matrix — Greenfield Skeleton Map

**Generated:** 2026-05-08 · **Status:** greenfield scaffold · **Stats:** 2442 files, 789 directories

This document is the orientation map for the freshly generated KFM repository
skeleton. It is intentionally expansive — every domain, register, contract,
schema, policy, validator, runbook, viewer surface, and connector mentioned
across the supplied dossiers has a visible home, even if just a stub
`README.md`. Trim later.

---

## 1. The seven planes

KFM is organized so that the **planes** are visible before the **topics**.
The planes overlap in topic but not in authority.

| Plane | What it owns | Where it lives |
|------|--------------|----------------|
| **Doctrine & architecture** | Why KFM exists; what `evidence-first`, `map-first`, `time-aware`, `policy-aware`, and `cite-or-abstain` mean. ADRs. | `docs/doctrine/`, `docs/architecture/`, `docs/adr/` |
| **Trust contracts & shape** | Object families with semantic meaning (contracts) and machine-checkable shape (schemas). | `contracts/`, `schemas/contracts/v1/` |
| **Policy & admissibility** | Allow/deny/restrict/abstain/redaction, sensitivity floors, rights compatibility, promotion gates. | `policy/` (singular) |
| **Lifecycle data** | RAW → WORK / QUARANTINE → PROCESSED → CATALOG / TRIPLET → PUBLISHED. Receipts, proofs, registries, manifests. | `data/` |
| **Implementation** | Apps, packages, connectors, pipelines, pipeline_specs, tools, scripts. | `apps/`, `packages/`, `connectors/`, `pipelines/`, `pipeline_specs/`, `tools/`, `scripts/` |
| **Release & correction** | Release decisions, manifests, signatures, rollback cards, correction notices, withdrawal notices. | `release/` |
| **Runtime & exposure** | Local runtime, model adapters, deployment, network, hardening. Deny-by-default at every public boundary. | `runtime/`, `infra/`, `configs/` |

Plus the `control_plane/` register set (machine-readable governance maps),
`tests/` and `fixtures/` (the proof that doctrine is enforceable),
`migrations/` (governed schema/data migrations), `examples/` (walkthroughs),
and `artifacts/` (compatibility / build / qa / temporary).

imports MapLibre runtime APIs.

## 3. Lifecycle invariant

```
RAW → WORK / QUARANTINE → PROCESSED → CATALOG / TRIPLET → PUBLISHED
```

Every transition is a **governed state change**, not a file move. The
artifacts that prove a transition happened live in:

| Stage | Authority artifact | Validator |
|-------|--------------------|-----------|
| Source admission | `data/registry/sources/<source>.yaml` + `data/receipts/ingest/` | `tools/validators/validate_source_descriptor.py` |
| RAW → WORK | `data/receipts/pipeline/` | `tools/validators/validate_run_receipt.py` |
| WORK → QUARANTINE | quarantine case record | per-domain validator |
| WORK → PROCESSED | `data/receipts/validation/` | `tools/validators/validate_evidence_bundle.py` |
| PROCESSED → CATALOG | `data/catalog/{stac,dcat,prov}/` | `tools/validators/validate_catalog_matrix.py` |
| Catalog → PROOF | `data/proofs/evidence_bundle/`, `data/proofs/proof_pack/` | `tools/proof_pack/assemble_proof_pack.py` |
| PROOF → REVIEW | `release/promotion_decisions/` | `tools/validators/validate_promotion_gate.py` |
| REVIEW → PUBLISHED | `release/manifests/` + `data/published/` | `tools/release/release_dry_run.py` |
| Rollback | `release/rollback_cards/` | `tools/release/rollback_apply.py` |
| Correction | `release/correction_notices/` + `release/withdrawal_notices/` | governed via review console |

## 5. Connector inventory (source families)

`connectors/` ships stubs for every source family seen across the dossiers.
Output goes to `data/raw/<domain>/<source_id>/<run_id>/` or
`data/quarantine/<domain>/<reason>/<run_id>/`. None of these connectors
publish; that path is governed.

`usgs · fema · noaa · nrcs · epa · blm · ahgp · khri · ksgs · kdwp ·
ksu_research_extension · kansas_state_archives · kansas_memory · loc ·
census · gbif · inaturalist · ebird · openstreetmap · newspapers ·

`fema_nfhl.yaml`, `nrcs_ssurgo.yaml`, `gbif.yaml`, `inaturalist.yaml`,

## 7. Reading order for a new contributor

1. `README.md` and `docs/doctrine/` — what KFM *is*.
2. `docs/architecture/SYSTEM_MAP.md` and `docs/architecture/DIRECTORY_RULES.md` — how it’s laid out.
3. `docs/adr/` — what’s already been decided.
4. `contracts/OBJECT_MAP.md` — the crosswalk from object → contract → schema → policy → fixture → validator → emit home.
5. `control_plane/` — the machine-readable registers.
6. `docs/domains/hydrology/` — the first proof-bearing lane.
7. `apps/governed-api/` — the trust membrane.
8. `apps/explorer-web/` — the map-first shell.
9. `pipelines/hydrology/` and `pipeline_specs/hydrology/` — the working *how* and *what*.
10. `release/` and `data/published/hydrology/` — what published actually looks like.

## 10. What to do next

This skeleton is *inventive scaffolding*. Suggested trim/promote sequence:

1. **Seal the doctrine layer.** Read `docs/doctrine/`, accept or rewrite,
   then promote with ADR-0001..0003.
2. **Seal the canonical homes.** Confirm `schemas/contracts/v1/` is
   canonical (ADR-0001), `contracts/` is semantic-only (ADR-0002),
   `policy/` is singular (ADR-0003).
3. **Pick the first proof slice.** The dossiers recommend hydrology;
   keep the rest of the domain stubs but defer their work.
4. **Build the trust membrane stubs first.** `apps/governed-api/`
   bootstrap + layers + evidence routes returning fixture-backed
   `DecisionEnvelope`s — even if they always return `ABSTAIN` at first.
5. **Land the deny suite early.** `tests/api/deny/` and
   `.github/workflows/deny-test.yml` should pass before any real
   public surface.
6. **Trim what you don't need.** Many of these stubs exist only to make
   the architecture *visible*. Delete freely.

Greenfield scaffolding is over-rich on purpose. The point is to not have
to ask "where would that go?" — every domain, family, gate, and viewer
already has a home.
