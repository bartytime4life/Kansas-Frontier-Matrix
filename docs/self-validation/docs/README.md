## üìò Overview

This directory stores all **documentation validation outputs** generated by CI/CD workflows and pre-commit hooks.  
Validation ensures that all Markdown, YAML, and JSON files across the repository follow:

- **MCP Documentation-First Protocol (v6.3)**
- **Platinum README Template v7.1**
- **FAIR+CARE documentation standards**
- **Consistent YAML front-matter schemas**

Validation occurs via the `.github/workflows/docs-lint.yml` workflow and is reinforced by **pre-commit hooks** for local enforcement.

## üßæ Report Descriptions

### 1. üß† `lint_summary.json`
Contains the summarized outcome of documentation validation runs.

| Field | Description |
|--------|-------------|
| `total_files` | Total number of documentation files checked. |
| `passed` | Number of files passing all linting rules. |
| `failed` | Number of files with one or more violations. |
| `rules_tested` | Total number of style or schema validation rules executed. |
| `timestamp` | UTC timestamp of the linting session. |

**Example:**
```json
{
  "total_files": 87,
  "passed": 87,
  "failed": 0,
  "rules_tested": 14,
  "timestamp": "2025-11-05T18:20:00Z"
}
```

---

### 2. üìã `violations.ndjson`
Raw NDJSON output listing each violation identified during Markdown, YAML, or JSON validation.

| Field | Description |
|--------|-------------|
| `file` | Path to the file with the issue. |
| `line` | Line number where the violation occurred. |
| `rule_id` | Linting rule or schema ID violated. |
| `severity` | warning / error. |
| `message` | Description of the issue detected. |

**Example:**
```json
{"file":"docs/README.md","line":25,"rule_id":"MD041","severity":"warning","message":"First line in file should be a top-level heading"}
{"file":"src/pipelines/README.md","line":12,"rule_id":"YAML_SCHEMA_MISSING_FIELD","severity":"error","message":"Missing required field 'version' in front-matter"}
```

## ‚öôÔ∏è Validation Workflows

**Workflow:** `.github/workflows/docs-lint.yml`

**Process Overview:**
1. Identify all Markdown, YAML, and JSON files in the repository.  
2. Validate Markdown structure and heading hierarchy (Markdownlint).  
3. Check YAML front-matter fields against the MCP schema.  
4. Validate JSON format and indentation consistency.  
5. Generate reports: `lint_summary.json` and `violations.ndjson`.  
6. Upload results as artifacts and append to:
   - `reports/audit/github-workflows-ledger.json`
   - `releases/v9.7.0/focus-telemetry.json`

## ‚öñÔ∏è FAIR+CARE Documentation Criteria

| Principle | Validation Target |
|------------|-------------------|
| **Findable** | Each document includes YAML metadata (`title`, `path`, `version`, `commit_sha`). |
| **Accessible** | Markdown follows open GFM syntax, CC-BY 4.0 license. |
| **Interoperable** | YAML schemas align with STAC/DCAT/DCMI metadata fields. |
| **Reusable** | Version and provenance recorded for every file. |
| **CARE** | Documentation avoids exclusionary or culturally insensitive language. |

## üß© Pre-Commit Hook Integration

Local developers must install pre-commit validation to ensure documentation passes checks before committing:

**Installation:**
```bash
pip install pre-commit
pre-commit install
```

**Execution Example:**
```bash
pre-commit run --all-files
```

**Rules Enforced Locally:**
- Markdownlint configuration: `.markdownlint.yaml`
- YAML schema validation: `.yamllint.yaml`
- Front-matter completeness
- Proper emoji usage in titles
- No trailing whitespace or missing EOF newline
