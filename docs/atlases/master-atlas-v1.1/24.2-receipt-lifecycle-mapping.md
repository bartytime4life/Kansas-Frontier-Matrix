## 1. Purpose and role

The receipt-class catalog (§24.2.1) tells you **what each receipt is**. The lifecycle-phase mapping (§24.2.2) tells you **where each receipt attaches** across the lifecycle invariant. The two are complementary; the catalog without the mapping leaves you guessing when to emit a `RedactionReceipt` vs. an `AggregationReceipt`; the mapping without the catalog leaves you with phase columns that don't explain what each row is for.

This file carries the mapping specifically because:

- The mapping is a single self-contained reference table that maintainers consult when wiring a new pipeline stage, a new validator, or a new release gate. It deserves a focused entry point.
- Atlas §24.2.2's *reading note* ("Receipts created earlier remain referenced (not duplicated) at later phases via EvidenceRef") is the operational rule that explains why receipts appear in multiple phases — that rule deserves prominence over inline placement in a 16-row catalog table.
- The matrix exposes structural patterns (cross-phase receipts vs. phase-specific receipts; admission-only receipts; release-only receipts) that the flat catalog ordering doesn't make visible. §7 surfaces these.

**This file is not authority.** Two non-collapse rules:

1. **Atlas v1.1 §24.2.2 wins on wording.** The 16-row × 5-column matrix is reproduced verbatim in §4.
2. **§24.2 wins over §24.2.2 for receipt definitions.** This file does not redefine `SourceDescriptor`, `TransformReceipt`, etc. — those definitions live in §24.2.1 and its carrier (`24.2-receipt-catalog.md`).

## 3. Lifecycle invariant

> **Source:** `directory-rules.md` lifecycle invariant; Atlas v1.0 §1 Operating Law; Atlas v1.1 §24.6.1 lifecycle gates.

Every domain in KFM follows the lifecycle invariant `RAW → WORK / QUARANTINE → PROCESSED → CATALOG / TRIPLET → PUBLISHED`. Promotion between phases is a **governed state transition**, not a file move.

| Phase | Role |
|---|---|
| **`RAW`** | Captured source payload (or reference) with admission-time identity. Immutable. |
| **`WORK / QUARANTINE`** | Normalized; validated; failures held. The two states are alternates: `WORK` if validation/policy pass; `QUARANTINE` if not. |
| **`PROCESSED`** | Validated normalized objects, receipts, and public-safe candidates emitted. |
| **`CATALOG / TRIPLET`** | Catalog records, EvidenceBundles, and graph/triplet projections emitted; release candidates assembled. |
| **`PUBLISHED`** | Released public-safe artifacts served through governed APIs and manifests. |

The five phase columns in §4 below correspond to these five lifecycle states.

> **Trust-membrane invariant:** the membrane between `WORK / QUARANTINE` and `PROCESSED` is where validation + policy gates close; the membrane between `CATALOG / TRIPLET` and `PUBLISHED` is where review + release close. No public client, normal UI surface, or released AI surface ever reaches `RAW`, `WORK`, or `QUARANTINE`. See `24.6-pipeline-gate-reference.md` (PROPOSED, flat-namespace twin `pipeline-gate-reference.md`).

### 4.1 Reading note

> **`CONFIRMED doctrine`** *(reproduced verbatim from §24.2.2)*: a dot means the receipt is normally emitted, amended, or referenced at that phase. Receipts created earlier remain **referenced (not duplicated)** at later phases via `EvidenceRef`.

This reading note is doctrinally central. Three consequences:

1. **A dot ≠ "emitted at this phase."** A dot means *"normally emitted, amended, **or referenced**."* A `SourceDescriptor` is emitted at admission (`RAW`) and **referenced** (not re-emitted) at every subsequent phase via `EvidenceRef`.
2. **`EvidenceRef` is the linkage mechanism.** Receipts are append-only; later-phase artifacts cite earlier-phase receipts by reference. The mapping shows where references resolve, not where new instances are created.
3. **Phase coverage is not duplication.** When a `RedactionReceipt` shows dots at `WORK / QUARANTINE`, `PROCESSED`, `CATALOG / TRIPLET`, and `PUBLISHED`, it means the same receipt instance is referenced from each of those phases — not that four copies exist.

## 5. Per-phase view: which receipts attach where

> Synthesis derived from §4. Each row inverts the matrix to answer the question *"At this phase, which receipts are normally emitted, amended, or referenced?"*

| Phase | Receipts normally present | Notes |
|---|---|---|
| **`RAW`** | `SourceDescriptor`, `PolicyDecision` | The thinnest phase. Admission is the only emission point; the policy decision is the admission decision. |
| **`WORK / QUARANTINE`** | `SourceDescriptor`, `TransformReceipt`, `RedactionReceipt`, `AggregationReceipt`, `ModelRunReceipt`, `ReviewRecord`, `PolicyDecision`, `ValidationReport` | The work phase. Most transform/normalization receipts are emitted here; validation gate closes here. |
| **`PROCESSED`** | `SourceDescriptor`, `TransformReceipt`, `RedactionReceipt`, `AggregationReceipt`, `ModelRunReceipt`, `RepresentationReceipt`, `ReviewRecord`, `PolicyDecision`, `ValidationReport`, `RealityBoundaryNote`, `MatrixCellReceipt` | The validated-output phase. Synthetic / 3D / matrix-cell content first appears here (with reality-boundary discipline). |
| **`CATALOG / TRIPLET`** | All except `RAW`-only receipts (so 15 of 16). | The catalog/proof-closure phase. `AIReceipt`, `ReleaseManifest`, `CorrectionNotice`, `RollbackCard`, `StorySnapshot` first appear here. |
| **`PUBLISHED`** | `SourceDescriptor`, `RedactionReceipt`, `AggregationReceipt`, `ModelRunReceipt`, `RepresentationReceipt`, `AIReceipt` *(Focus Mode only)*, `ReviewRecord`, `PolicyDecision`, `ReleaseManifest`, `CorrectionNotice`, `RollbackCard`, `RealityBoundaryNote`, `MatrixCellReceipt`, `StorySnapshot` | The public-surface phase. `TransformReceipt` and `ValidationReport` are **not** referenced here (their work is upstream and complete). |

### 5.1 What's missing at `PUBLISHED` (and why)

> `TransformReceipt` and `ValidationReport` are the two receipts that **do not** show a dot at `PUBLISHED`. This is intentional and operationally significant.

| Receipt | Why absent at `PUBLISHED` |
|---|---|
| `TransformReceipt` | Records transforms applied during normalization (reprojection, generalization, snap). Once `PROCESSED`, the transform is fixed; subsequent phases reference its **output**, not its **execution record**. `TransformReceipt` is upstream-only. |
| `ValidationReport` | Records validator runs against the working set. Once a feature has passed validation and entered `PROCESSED`, its `ValidationReport` is closed; later phases rely on `PolicyDecision` and `ReleaseManifest` for gate decisions, not on the upstream validator report. |

Both receipts remain **retrievable** at `PUBLISHED` via `EvidenceRef` (the bundle resolution chain reaches back through `CATALOG / TRIPLET`), but they are not normally **referenced from** the published artifact.

## 6. Per-receipt view: which phases each receipt spans

> Synthesis derived from §4. Inverts the matrix to answer the question *"For this receipt, which phases is it normally present at?"*

| Receipt | Phase span | Pattern |
|---|---|---|
| `SourceDescriptor` | `RAW` → `PUBLISHED` (all 5) | **Cross-phase anchor.** Always referenced; never re-emitted after admission. |
| `PolicyDecision` | `RAW` → `PUBLISHED` (all 5) | **Cross-phase governance.** Each phase emits its own (Admission, Normalization, Validation, Release, Correction). |
| `TransformReceipt` | `WORK / QUARANTINE` → `CATALOG / TRIPLET` (3 phases) | **Mid-pipeline.** Emitted during normalization; absent at `RAW` and `PUBLISHED`. |
| `ValidationReport` | `WORK / QUARANTINE` → `CATALOG / TRIPLET` (3 phases) | **Mid-pipeline.** Same span as `TransformReceipt`. Closed when validator passes; not referenced at `PUBLISHED`. |
| `RedactionReceipt` | `WORK / QUARANTINE` → `PUBLISHED` (4 phases) | **Sensitivity pipeline.** Sensitivity transforms can apply at any phase from work onward; the receipt persists into publication. |
| `AggregationReceipt` | `WORK / QUARANTINE` → `PUBLISHED` (4 phases) | **Aggregate pipeline.** Same span as `RedactionReceipt`. |
| `ModelRunReceipt` | `WORK / QUARANTINE` → `PUBLISHED` (4 phases) | **Modeling pipeline.** Same span as `RedactionReceipt`. |
| `ReviewRecord` | `WORK / QUARANTINE` → `PUBLISHED` (4 phases) | **Review pipeline.** Reviewers act at multiple phases (admission review, redaction review, release review). |
| `RepresentationReceipt` | `PROCESSED` → `PUBLISHED` (3 phases) | **Representation pipeline.** 3D scenes / tile representations first appear at `PROCESSED`. |
| `RealityBoundaryNote` | `PROCESSED` → `PUBLISHED` (3 phases) | **Synthetic-discipline pipeline.** Paired with `RepresentationReceipt`. |
| `MatrixCellReceipt` | `PROCESSED` → `PUBLISHED` (3 phases) | **Frontier Matrix pipeline.** Same span as `RepresentationReceipt` and `RealityBoundaryNote`. |
| `ReleaseManifest` | `CATALOG / TRIPLET` → `PUBLISHED` (2 phases) | **Release pipeline.** Assembled at catalog closure; referenced at publication. |
| `CorrectionNotice` | `CATALOG / TRIPLET` → `PUBLISHED` (2 phases) | **Correction pipeline.** Same span as `ReleaseManifest`. |
| `RollbackCard` | `CATALOG / TRIPLET` → `PUBLISHED` (2 phases) | **Rollback pipeline.** Same span. |
| `StorySnapshot` / `ExportReceipt` | `CATALOG / TRIPLET` → `PUBLISHED` (2 phases) | **Export pipeline.** Same span. |
| `AIReceipt` | `CATALOG / TRIPLET` → `PUBLISHED` (Focus Mode only) (2 phases) | **AI surface pipeline.** Latest entry into the lifecycle; restricted at `PUBLISHED` to Focus Mode only. |

## 7. Emission patterns observed in the matrix

> Three structural patterns emerge from the §4 matrix that are not visible in the receipt-class catalog alone.

### 7.1 Cross-phase receipts (the spine)

| Receipt | Span | Role |
|---|---|---|
| `SourceDescriptor` | All 5 phases | Anchors every downstream receipt via `EvidenceRef`. |
| `PolicyDecision` | All 5 phases | Governance at every transition. |

These two receipts are **the spine** of the lifecycle. Every artifact at every phase carries a `SourceDescriptor` reference and a `PolicyDecision` reference. They are the source-role identity attribute (`SourceDescriptor.source_role`, see **`24.1-source-role-anti-collapse.md`**) and the deny-or-allow attribute (`PolicyDecision`).

### 7.2 Pipeline-bound receipts (the work)

Most receipt classes span 2–4 phases starting at `WORK / QUARANTINE` or `PROCESSED` and ending at `CATALOG / TRIPLET` or `PUBLISHED`. They record work done during the pipeline. Examples:

- **Mid-pipeline (3 phases):** `TransformReceipt`, `ValidationReport`, `RepresentationReceipt`, `RealityBoundaryNote`, `MatrixCellReceipt`.
- **Wide-pipeline (4 phases):** `RedactionReceipt`, `AggregationReceipt`, `ModelRunReceipt`, `ReviewRecord`.

The asymmetry tells you which receipts persist all the way to public surfaces (`RedactionReceipt`, `AggregationReceipt`, `ModelRunReceipt`, `ReviewRecord`) and which complete their work before publication (`TransformReceipt`, `ValidationReport`).

### 7.3 Late-stage receipts (the release surface)

| Receipt | Span | Role |
|---|---|---|
| `ReleaseManifest` | `CATALOG / TRIPLET` + `PUBLISHED` | Bound contents + version + signatures + rollback target. |
| `CorrectionNotice` | `CATALOG / TRIPLET` + `PUBLISHED` | Post-publication correction lineage. |
| `RollbackCard` | `CATALOG / TRIPLET` + `PUBLISHED` | Rollback decision and target. |
| `StorySnapshot` / `ExportReceipt` | `CATALOG / TRIPLET` + `PUBLISHED` | Story / export publication state. |
| `AIReceipt` | `CATALOG / TRIPLET` + `PUBLISHED` *(Focus Mode only)* | Governed AI answer receipt. |

These five receipts share a 2-phase span at the end of the lifecycle. They are the **release-surface receipts** — the records that make a publication auditable, correctable, and reversible.

> **The pattern is the policy.** The matrix's structural patterns mirror the operating-law discipline: a release without a `ReleaseManifest` + correction path + rollback target is not a governed release. The phase mapping makes that requirement visible.

### 7.4 The `AIReceipt` exception

`AIReceipt` is unique in two ways:

- It has the **latest entry** into the lifecycle (no dot until `CATALOG / TRIPLET`).
- Its `PUBLISHED` dot carries the qualifier `(Focus Mode only)`.

Both reflect the cite-or-abstain doctrine: AI surfaces never operate on raw or working data; they only operate on released `EvidenceBundle` content via Focus Mode. The published AI surface is governed by `AIReceipt`, which itself is governed by the `PolicyDecision` and `ReviewRecord` at the same phase.
