# KFM Project Source Ledger

> [!IMPORTANT]
> This is a **source-routing ledger**, not a bibliography. It tells future KFM work which files to consult, what each can support, and what each must not be used to prove.

## Purpose

Create a durable human-readable and machine-readable index for the KFM Drive/source corpus so future AI, documentation, architecture, repo, and research work can route claims to the right evidence and avoid treating doctrine, lineage, exploratory ideas, references, and implementation proof as interchangeable.
## Current truth posture

| Label | Current determination |
|---|---|
| CONFIRMED | Current request, uploaded/local PDFs, page counts/sizes for local PDFs, accessible snippets from KFM documentation/control-plane sources, and Google Drive metadata returned by search/recent-document tools. |
| PROPOSED | Source classifications, authority rank assignments where no canonical register was directly inspected, routing rules, related-source mappings, and recommended file paths. |
| UNKNOWN | Full contents of Drive-only files not fetched in this run, exact owner/author for many PDFs, repository implementation maturity, route names, schemas, tests, workflows, dashboards, logs, deployed behavior, rights, and public-release safety. |
| NEEDS_VERIFICATION | Source rights, licensing, endpoints, current versions, official-source terms, source authority, duplicate/supersession relationships, and any claim that a PDF plan exists as implemented repo behavior. |

## Source authority model

| Rank | Source class | Use | Warning |
|---:|---|---|---|
| 0 | Current user request / controlling instruction | Controls this ledger task and immediate output rules. | Does not become permanent doctrine by itself. |
| 1 | Mounted repo evidence, tests, workflows, schemas, logs, emitted artifacts | Proves current implementation behavior when directly inspected. | No mounted repo evidence was available in this run. |
| 2 | Existing canonical source ledger or documentation-control register | Controls source status and supersession if directly inspected. | Use KFM documentation architecture/living manual as best available until repo register exists. |
| 3 | Current KFM doctrine, master manuals, living manuals, synthesis passes | Governs KFM meaning and architecture. | Does not prove current implementation. |
| 4 | Subsystem architecture manuals | Governs MapLibre, Cesium, governed AI, pipeline, API/UI boundaries. | Routes/tests/package pins remain UNKNOWN without repo evidence. |
| 5 | Domain-lane architecture blueprints | Governs lane semantics and risk posture. | Domain plans do not prove files/connectors exist. |
| 6 | New Ideas packets and exploratory sketches | Idea intake and backlog. | Never cite as doctrine unless promoted by stronger evidence. |
| 7 | External official standards/data-source docs | Current versions, endpoints, licenses, standards. | Must be freshly checked for version-sensitive work. |
| 8 | General technical/GIS/database/API/UI references | Background concepts and implementation literacy. | Do not define KFM policy. |
| 9 | Memory, assumptions, informal notes | Not authoritative. | Do not cite as evidence. |

## Supersession and lineage notes

- Older broad KFM manuals, implementation guides, package documents, and early architecture PDFs are **retained as LINEAGE** until a canonical register explicitly supersedes them.
- Drive duplicates with identical or near-identical titles are **not merged silently**. They require content hash comparison and successor-link decisions.
- New Ideas packets are **EXPLORATORY** by default. They become doctrine only through a governed intake decision, schema/contract update if applicable, validation, and review.
- Domain-lane PDF-only blueprints are implementation-grade planning artifacts, but they **do not prove repo files exist**.
- General references are retained because they support GIS, SQL, API, React/TypeScript, CI/CD, AI, statistics, and temporal modeling literacy; they do **not** define KFM governance.

## Maintenance rules

1. Never upgrade a source from `EXPLORATORY`, `LINEAGE`, or `REFERENCE` to doctrine without an explicit authority decision.
2. Never cite a PDF plan as proof of repo implementation.
3. Unknown rights or unresolved sensitivity blocks public promotion.
4. Keep receipts, proofs, catalogs, release manifests, review records, correction notices, and source descriptors separate.
5. For current behavior, mounted repo evidence outranks PDFs.
6. For doctrine, KFM current manuals and documentation-control registers outrank general technical references.
7. Every new source must have `what_it_does_not_prove` filled before it can be cited by future AI work.

## Update procedure

1. Add or update the source in `data/registry/sources/project_source_ledger.yaml`.
2. Fetch/export Drive-only files and record page count, file hash, owner/author if available, created/modified dates, and rights posture.
3. Assign family, authority rank, status, domains, object families, and source-use rules.
4. Update domain and object-family matrices.
5. Add supersession links rather than deleting older lineage.
6. Run a ledger completeness validator once available.
7. Update this Markdown from the YAML or manually reconcile both files.
8. Record unresolved items in the verification backlog.
