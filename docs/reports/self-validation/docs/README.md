## üìò Overview

This directory contains automated **documentation validation reports** that verify:
- Proper YAML front-matter metadata  
- Valid Markdown structure and section hierarchy  
- Correct table formatting and heading levels  
- FAIR+CARE-compliant language and inclusivity standards  

All results are generated by the CI workflow:  
`.github/workflows/docs-lint.yml`

and are automatically cross-linked to:
- `reports/audit/github-workflows-ledger.json` (validation events)
- `releases/v9.7.0/focus-telemetry.json` (telemetry dashboard)
- `docs/reports/telemetry/governance_scorecard.json` (FAIR+CARE documentation metrics)

## üßæ Report Details

### 1. üßÆ `lint_summary.json`
Contains a summary of linting and formatting checks across all documentation files.

| Field | Description | Example |
|--------|-------------|----------|
| `total_files` | Number of documentation files validated. | `87` |
| `passed` | Files with zero violations. | `87` |
| `failed` | Files with errors or warnings. | `0` |
| `rules_tested` | Count of linting or schema checks executed. | `14` |
| `timestamp` | UTC timestamp of validation. | `"2025-11-05T18:20:00Z"` |

**Example:**
```json
{
  "total_files": 87,
  "passed": 87,
  "failed": 0,
  "rules_tested": 14,
  "timestamp": "2025-11-05T18:20:00Z"
}
```

---

### 2. üßæ `violations.ndjson`
Raw NDJSON stream containing all detected rule violations for auditing and review.

| Field | Description | Example |
|--------|-------------|----------|
| `file` | File path where the violation occurred. | `"docs/README.md"` |
| `line` | Line number of the issue. | `25` |
| `rule_id` | Linting rule or schema ID violated. | `"MD041"` |
| `severity` | `warning` or `error`. | `"warning"` |
| `message` | Human-readable description of issue. | `"First line in file should be a top-level heading."` |

**Example Entry:**
```json
{"file":"docs/README.md","line":25,"rule_id":"MD041","severity":"warning","message":"First line should be a top-level heading"}
```

## ‚öôÔ∏è Validation Workflow Integration

**Workflow:** `.github/workflows/docs-lint.yml`

**Pipeline Stages:**
1. Detect all Markdown (`.md`), YAML (`.yml`), and JSON files.  
2. Validate front-matter presence and schema correctness.  
3. Enforce Markdown rules (MD041, MD022, MD025, etc.).  
4. Verify internal link integrity and table alignment.  
5. Check file encoding and newline consistency.  
6. Generate and upload validation artifacts.  

**Output Artifacts:**
- `lint_summary.json`  
- `violations.ndjson`  
- Telemetry update in `focus-telemetry.json`

## üß† Common Validation Rules

| Rule ID | Category | Description |
|----------|-----------|-------------|
| `MD041` | Heading | First line must be a top-level heading. |
| `MD022` | Formatting | Headings must be surrounded by blank lines. |
| `MD025` | Structure | Only one H1 per document. |
| `MD040` | Code Blocks | Fenced code blocks must have a language specifier. |
| `MD033` | HTML | Inline HTML elements must be closed properly. |
| `MD009` | Whitespace | Trailing spaces not allowed. |
| `YAML_SCHEMA_MISSING_FIELD` | Metadata | Required front-matter field missing. |
| `LINK_CHECK_FAILURE` | Links | Broken or unresolved internal link. |
