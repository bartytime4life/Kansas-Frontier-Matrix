## 1. Purpose

The durable question for this lane is:

> Can a governed Archaeology process transform an already admitted, immutable source capture or controlled WORK record into a validation-ready candidate while preserving original meaning, source role, candidate status, time kinds, spatial precision, rights, cultural and sovereignty obligations, evidence dependencies, correction lineage, and rollback support—without exposing protected information or silently manufacturing certainty?

A mature normalizer may eventually:

- read only admitted immutable inputs or synthetic public-safe fixtures;
- apply an accepted, versioned, digest-pinned mapping profile;
- preserve original fields and record every mapping, default, coercion, omission, and transform;
- normalize object-family hints without promoting candidates to confirmed sites;
- separate observations, administrative records, archival context, models, interpretations, aggregates, synthetic records, and restricted material;
- preserve observed, recorded, valid, survey, collection, accession, processing, review, release, correction, and supersession times where material;
- preserve internal geometry and derive only explicitly classified representation candidates;
- calculate deterministic candidate identity and content digests under accepted contracts;
- emit a typed normalization result or route the input to `QUARANTINE`, `HOLD`, `ABSTAIN`, `DENY`, or `ERROR`;
- preserve correction, revocation, withdrawal, supersession, and rollback dependencies.

This lane must not:

- fetch or activate sources;
- decide source admission;
- define Archaeology object meaning or machine shape;
- resolve cultural, sovereignty, consent, rights-holder, or steward authority;
- treat mapping success as evidence closure or validation success;
- declare a representation public-safe merely because coordinates were generalized;
- store lifecycle data, receipts, proofs, policies, review records, or releases beside code;
- expose exact or triangulable protected locations;
- write release manifests, public maps, public APIs, search indexes, exports, screenshots, or AI answers.

## 3. Normalization anti-collapse rules

A future implementation must preserve these distinctions:

```text
admitted source capture            != normalized WORK candidate
normalized WORK candidate          != validated or processed object
candidate feature                  != confirmed archaeological site
remote-sensing anomaly             != field observation
schema-shaped record               != valid evidence
mapping completion                 != source-role correctness
field alias                        != semantic equivalence
parsed date                        != known event time
coordinate conversion              != public-safe geometry
geometry repair                    != permission to disclose
generalized geometry               != cultural or steward approval
EvidenceRef candidate              != EvidenceBundle
transform obligation               != TransformReceipt
review reference                   != current valid review
policy reference                   != PolicyDecision
normalization receipt              != ValidationReport or ProofPack
workflow success                   != normalization, validation, release, or publication
generated summary                  != evidence or cultural authority
merge                              != lifecycle promotion
```

Required invariants:

1. **Original values remain recoverable.** Normalization must not erase source wording, units, dates, geometry references, nulls, qualifiers, uncertainty, or provenance.
2. **Every mutation is explainable.** Each mapped, coerced, defaulted, dropped, redacted, repaired, or derived field has a method or mapping reference.
3. **Candidate status is monotonic until governed change.** Labels, scores, normalized types, map styling, graph edges, or prose cannot confirm a site.
4. **Source role survives every transformation.** Observation, archival, administrative, regulatory, modeled, interpreted, aggregate, synthetic, candidate, contextual, and restricted records remain distinguishable.
5. **Time kinds remain separate.** Unknown, approximate, inferred, bounded, source, observation, valid, processing, review, release, and correction times are not silently flattened.
6. **Sensitivity never decreases by convenience.** Coordinate rounding, hashing, aggregation, or generalization is not automatically a public-safe transformation.
7. **Human authority remains explicit and current.** Cultural, sovereignty, consent, revocation, embargo, rights-holder, and steward states cannot be inferred.
8. **Lifecycle state remains explicit.** RAW, WORK, QUARANTINE, PROCESSED, CATALOG/TRIPLET, release candidate, and PUBLISHED are separate states.
9. **Correction and rollback dependencies remain attached.** A downstream candidate without source version, mapping version, and invalidation relationships is not reproducible.

## 7. Normalization input boundary

A future normalization run must refuse or hold an input unless the accepted profile can establish:

- immutable input reference and digest;
- admitted lifecycle state;
- source identity, family, role, version, vintage, rights, access, and sensitivity posture;
- source-intake or RAW-capture receipt reference;
- object-family hint and candidate/confirmed status;
- original field names, values, units, qualifiers, null semantics, and encoding;
- temporal values with source precision and uncertainty;
- spatial reference, coordinate reference system, geometry precision, and access class where geometry is present;
- cultural, sovereignty, consent, revocation, embargo, steward, and rights-holder obligations where applicable;
- mapping-profile identity, version, digest, owner, and compatibility constraints;
- expected output contract and schema version;
- correction, supersession, and rollback references for reprocessing.

|---|---|---|

### Untrusted input posture

Treat source strings, markup, OCR, filenames, archive paths, embedded metadata, generated text, and external identifiers as untrusted data. They cannot alter instructions, select schemas, widen permissions, activate network access, choose release state, or expose secrets.

## 8. Mapping, original-field, and identity obligations

### Lossless mapping ledger

Every output field should be traceable to one or more source fields or an explicit governed derivation.

| Mapping property | Minimum obligation |
|---|---|
| Source field | Original key/path and source value or immutable reference. |
| Output field | Target field and schema version. |
| Operation | copy, parse, map, coerce, derive, aggregate, redact-candidate, omit, or error. |
| Method | Mapping rule ID and implementation/spec digest. |
| Uncertainty | Source precision, confidence, qualifiers, and ambiguity. |
| Review | Whether human, cultural, rights-holder, or steward review is required. |
| Sensitivity | Input and candidate output classifications; no silent downgrade. |
| Time | Source and normalized time kinds plus timezone/precision handling. |
| Geometry | CRS, precision, repair, generalization obligation, and restricted/public representation class. |
| Reason | Structured reason for defaults, omissions, holds, quarantine, or denial. |

### Deterministic identity

Identity logic must be accepted before implementation. A candidate identity should be based on stable source and domain facts, not display labels, current timestamps, map zoom, AI output, or mutable file paths.

The implementation must distinguish:

- source-record identity;
- normalized-candidate identity;
- domain-entity identity;
- geometry/version identity;
- mapping-profile identity;
- run identity;
- receipt identity.

No one identity may silently substitute for another.

### Temporal handling

Normalization must preserve material time kinds rather than forcing every value into one timestamp:

```text
source time
observation / survey / collection time
recorded / accession time
valid-from / valid-to
retrieval / ingestion time
normalization / validation time
review / consent / revocation / embargo time
release / correction / withdrawal / rollback time
```

Unknown, approximate, inferred, range-bounded, and disputed values remain explicit.

### Geometry and representation

Normalization may:

- validate CRS declarations;
- preserve exact restricted geometry in an authorized internal reference;
- compute geometry fingerprints under accepted rules;
- prepare public-representation obligations;
- flag invalid, ambiguous, unsupported, or excessive precision.

Normalization must not:

- place protected geometry in logs or receipts;
- decide that a generalized geometry is culturally approved;
- expose transformation offsets or reversible hints;
- write public tiles, map sources, screenshots, search indexes, or exports;
- silently repair geometry without a mapping/transform record.

## 9. Lifecycle and finite result contract

### Lifecycle boundary

```text
admitted RAW / governed WORK
        |
        v
normalization preflight
        |
        +--> QUARANTINED / HOLD / ABSTAIN / DENY / ERROR
        |
        v
validation-ready WORK candidate
        |
        v
separate validation -> processed -> catalog/triplet -> release -> published
```

Normalization is a WORK-stage transformation. It does not promote directly to PROCESSED, CATALOG/TRIPLET, a release candidate, or PUBLISHED.

### Proposed result classes

| Result | Meaning | Permitted side effect |
|---|---|---|
| `NORMALIZED` | Accepted input was mapped losslessly enough for validation. | Write a governed WORK candidate and receipt through accepted adapters. |
| `QUARANTINED` | Input or transform is malformed, unsafe, restricted, conflicted, or unresolved. | Write a structured quarantine reference and safe receipt. |
| `HOLD` | A required review, authority, mapping, schema, rights, or policy dependency is pending. | Preserve state without higher lifecycle promotion. |
| `ABSTAIN` | Mapping cannot support the requested interpretation without guessing. | Record bounded non-result and reasons. |
| `DENY` | Policy or sensitivity prohibits the attempted processing or output. | No prohibited derivative; record safe reasons. |
| `ERROR` | Infrastructure, parser, schema, or integrity failure prevents trustworthy completion. | Fail closed; no partial unreceipted write. |

These are design-level classes until an accepted contract defines the machine enum.

### Transaction expectations

A mature run should behave atomically at the logical result level:

- no candidate write without its receipt;
- no partial field set presented as complete;
- no successful result if mandatory mappings or digests failed;
- no quarantine without a structured reason and input reference;
- no overwrite of prior immutable inputs or receipts;
- no hidden fallback from `DENY`, `HOLD`, `ABSTAIN`, or `ERROR` to `NORMALIZED`.

## 10. Sensitive-domain normalization gates

A future implementation must fail closed on every material gate.

| Gate | Minimum question | Fail-closed result |
|---|---|---|
| Input admission | Is the input admitted, immutable, and digest-pinned? | `DENY` or `QUARANTINED` |
| Spec binding | Is the normalize spec accepted, versioned, parsed, and bound to this consumer? | `ERROR` or `HOLD` |
| Mapping binding | Is the mapping profile accepted, compatible, owned, and digest-pinned? | `ERROR` or `HOLD` |
| Source role | Is role explicit and preserved? | `QUARANTINED` or `ABSTAIN` |
| Candidate boundary | Could any mapping promote a candidate, anomaly, or model to a confirmed site? | `DENY` |
| Original-field preservation | Can original values and qualifiers be reconstructed? | `QUARANTINED` |
| Object-family mapping | Is the target family supported without semantic guesswork? | `ABSTAIN` or `HOLD` |
| Time-kind preservation | Are material times, precision, uncertainty, and intervals explicit? | `QUARANTINED` or `ABSTAIN` |
| Geometry | Is protected precision contained and public representation only an obligation? | `DENY` or `QUARANTINED` |
| Rights and attribution | Are permitted transformations and obligations explicit? | `DENY` or `HOLD` |
| Cultural and sovereignty review | Are required authority and review states current and scoped? | `DENY` or `HOLD` |
| Consent, revocation, embargo | Are current controls present and unrevoked? | `DENY` |
| Evidence dependencies | Are source and candidate evidence refs preserved without being promoted to proof? | `ABSTAIN` or `HOLD` |
| Determinism | Are canonicalization, ordering, timezone, locale, seed, and digest behavior fixed? | `ERROR` |
| Output safety | Do candidate, receipt, log, and error surfaces avoid protected disclosure? | `DENY` or `ERROR` |
| Correction and rollback | Can downstream consumers be invalidated by source or mapping changes? | `HOLD` |

### Protected-output prohibition

Do not expose protected content in:

- process arguments;
- filenames or branch names;
- logger fields;
- exception messages;
- reason-code details;
- diffs and patches;
- test IDs or snapshots;
- workflow summaries and artifacts;
- receipts or manifests;
- telemetry labels;
- map, API, search, graph, embedding, export, screenshot, or AI surfaces.

Safe diagnostics identify the gate and reason family, not the protected value.

## 12. File admission and smallest sound implementation sequence

### File-admission rule

Do not add an executable merely because the v0.1 README proposed a filename. Admit a file only when:

1. its single primary responsibility is clear;
2. the owner and reviewer set are named;
3. input and output contracts are accepted;
4. the declarative specification is accepted and consumer-bound;
5. shared-versus-domain ownership is resolved;
6. no parallel mapping, schema, policy, fixture, receipt, or lifecycle authority is created;
7. no-network positive and negative fixtures exist;
8. deterministic behavior and safe diagnostics are testable;
9. correction and rollback are defined;
10. the change is small and reversible.

### Proposed sequence

1. **Resolve ownership.** Decide shared normalization mechanics versus Archaeology adapters.
2. **Accept semantic boundaries.** Define candidate, site, source role, object family, original-field, time-kind, and representation meanings.
3. **Accept schemas.** Define normalization input, mapping profile, mapping ledger, result, receipt, and reason-code shapes without parallel schema homes.
4. **Graduate the specification.** Replace `normalize.spec.yaml` placeholder only after parser, registry, owner, consumer, schema, and digest rules exist.
5. **Create synthetic fixtures.** Add compact public-safe accepted and fail-closed cases, including canary fields.
6. **Build a no-network runner.** Process one fixture deterministically without live source, model, geocoder, tile, database, or secret access.
7. **Add lossless mapping.** Prove original values, time kinds, roles, candidate state, geometry class, and mapping ledger.
8. **Add quarantine and finite results.** Make every unsupported or unsafe branch explicit and receipted.
9. **Add validation handoff.** Produce only a validation-ready WORK candidate; do not create `ValidationReport` or PROCESSED output.
10. **Wire CI deliberately.** Replace readiness holds only after accepted commands and meaningful tests exist.
11. **Run correction and rollback drills.** Change a source or mapping version and prove invalidation/recompile behavior.
12. **Seek human review.** Documentation, code, and tests do not grant cultural, policy, evidence, or release authority.

### Suggested future topology

The following is **PROPOSED**, not a current inventory:


Before any file is created, compare it with shared package, pipeline, validator, schema, and receipt homes.

### Default posture

- synthetic or explicitly public-safe;
- deterministic;
- no network;
- no credentials;

## 14. Correction, supersession, and rollback

Normalization outputs are derivatives. They must be invalidated when any trust-bearing dependency changes materially.

### Invalidation triggers

- source payload or digest correction;
- SourceDescriptor role, rights, attribution, sensitivity, or status change;
- consent revocation, embargo, cultural-review, sovereignty, or rights-holder change;
- mapping-profile correction or supersession;
- schema or contract incompatibility;
- object-family or candidate-status correction;
- time-kind, unit, vocabulary, CRS, geometry, or precision correction;
- policy decision or sensitivity-transform change;
- discovered protected-output leak;
- deterministic replay mismatch.

### Required propagation

A mature system should:

1. identify affected normalization results and downstream consumers;
2. mark results stale, withdrawn, blocked, or superseded;
3. prevent new validation or release use;
4. invalidate or rebuild dependent processed, catalog, graph, tile, search, export, and AI derivatives;
5. preserve prior immutable receipts and lineage;
6. emit correction and recompile records through accepted homes;
7. verify rollback through governed public surfaces.

### Documentation rollback

This README can be reverted through a normal reviewed Git revert. Reverting documentation does not roll back lifecycle data, source activation, policy, cultural review, evidence, release, or publication.
