## 1. Purpose

A future Agriculture pipeline may transform admitted Agriculture inputs into governed work candidates, quarantine records, processed derivatives, catalog/triplet handoffs, receipts, and release-review packages.

Its durable question is:

> Can an admitted, versioned, rights-resolved, source-role-preserving Agriculture input be transformed deterministically into a bounded Agriculture candidate or derivative while preserving identity, units, time, spatial support, uncertainty, privacy, evidence, policy, correction, and rollback obligations?

The lane may eventually support:

- aggregate crop and cropland observations;
- crop progress and condition records at the source-supported geography;
- yield, production, and economic aggregates with suppression state preserved;
- candidate field or management-unit geometry for restricted review only;
- irrigation, conservation, and land-use relationship candidates;
- Soil–Agriculture suitability interpretations that preserve Soil authority;
- atmosphere, drought, hazard, vegetation, and pest stress indicators with explicit knowledge character;
- catalog and graph handoffs that retain provenance and do not replace domain truth;
- public-safe aggregate products only after independent release approval.

This lane does not define what those objects mean, decide which sources are admitted, determine what policy allows, create evidence, approve a release, or serve a public client.

## 4. Anti-collapse rules

The following equivalences are denied:

```text
README or path                  -> implemented pipeline
spec file                       -> active execution
connector README                -> admitted source
source filename                 -> source role
aggregate statistic             -> field observation
remote-sensing pixel            -> crop or management truth
geometry similarity             -> field identity
parcel adjacency                -> operator ownership
modeled suitability             -> observed suitability
stress indicator                -> crop loss, emergency, or diagnosis
schema validity                 -> semantic validity
validator pass                  -> evidence closure
aggregation request             -> privacy-safe output
AggregationReceipt-shaped data  -> policy approval
pipeline success                -> processed promotion
processed state                 -> catalog closure
catalog/triplet entry           -> public release
workflow success                -> validation, proof, or release
commit or merge                 -> lifecycle promotion
generated summary               -> evidence
```

Required separations:

- native and normalized values;
- observed, regulatory, modeled, aggregate, administrative, candidate, and synthetic knowledge characters;
- Agriculture-owned facts and Soil-, Hydrology-, Atmosphere-, Hazards-, Habitat-, Geology-, People/Land-, Infrastructure-, Flora-, and Fauna-owned context;
- object identity and geometry similarity;
- source observation time, valid time, reporting period, retrieval time, processing time, release time, and correction time;
- transformation receipt, validation result, policy decision, EvidenceBundle, release record, and public artifact;
- internal exact or restricted views and released public-safe derivatives.

## 5. Agriculture pipeline scope

### Object and product families

The repository's Agriculture documentation names twelve semantic families. This pipeline may eventually process their candidates only through accepted contracts and schemas.

| Family | Future pipeline role | Must not become |
|---|---|---|
| `CropObservation` | Normalize crop, geography, period, units, source role, and uncertainty. | Unsupported field observation or current crop truth. |

| `YieldObservation` | Normalize geography, period, unit, suppression, and aggregate status. | Farm/operator yield claim. |
| `IrrigationLink` | Preserve an explicitly supported Agriculture–Hydrology relation. | Water right, ownership, permit, or operational instruction. |
| `ConservationPractice` | Normalize program/practice records and effective time. | Proof of current on-ground implementation. |
| `SoilCropSuitability` | Apply an accepted interpretation while retaining Soil references and scale limits. | Agriculture-owned Soil truth or prescription. |
| `AgriculturalEconomyObservation` | Preserve measure, geography, period, confidentiality, and suppression. | Private-party financial inference. |

### Domain-owned versus cross-domain work

Single-domain Agriculture transformations may belong here. Reusable cross-domain join validation and semantic relations remain in their lowest common responsibility roots.

A transformation belongs here only when Agriculture remains the clear owner of the derived object. Where ownership is genuinely shared or neutral, the implementation needs a reviewed cross-domain placement decision rather than silently picking Agriculture.

## 6. Source and input admission

### No source is activated here

The v0.1 README names NASS, NRCS, Mesonet, NOAA, NASA, and local/steward-curated source families. Those names are discovery context, not current admission evidence.

A future run may consume a source only when the following resolve:

- stable `SourceDescriptor` identity and version;
- explicit activation state and permitted operation;
- source role and knowledge character;
- rights, confidentiality, attribution, and redistribution terms;
- source-native units, IDs, geography, time, suppression, and quality flags;
- freshness and stale-state profile;
- sensitivity and re-identification posture;
- approved fixture and negative test coverage;
- connector/RAW lineage and immutable hashes;
- correction, withdrawal, and supersession handling.

### Allowed lifecycle inputs

| Input class | Future condition |
|---|---|
| Synthetic/no-network fixture | Reviewed, minimized, deterministic, public-safe, and bound to a named test. |
| Immutable RAW capture | Admitted source, checksums, intake receipt, rights posture, and no mutation. |
| WORK candidate | Explicit candidate status, owning producer, method, and blockers. |
| QUARANTINE record | Remediation-only operation with reviewer authorization and no public side effect. |
| PROCESSED baseline | Immutable version, validation refs, evidence posture, and correction state. |
| Cross-domain context ref | Stable owning-domain ref with source role, scale, time, sensitivity, and evidence preserved. |
| Policy/contract/schema ref | Accepted version and digest; referenced rather than copied. |
| Prior release | Used only for diff, correction, withdrawal, or rollback under release authority. |

### Denied input behavior

The pipeline must not continue silently when inputs have:

- unknown source identity or role;
- rights or confidentiality uncertainty;
- field/operator/parcel precision outside authorized scope;
- incompatible units, geography, time, or scale;
- missing suppression flags or no-data semantics;
- stale, corrected, withdrawn, superseded, or unversioned state;
- unresolved cross-domain ownership;
- missing EvidenceRef support where the operation is consequential;
- unaccepted policy or schema versions;
- secret, credential, token, or private endpoint material.

| Agriculture | Agriculture object families and Agriculture-owned derivatives. | May create candidates under accepted contracts. |
| Soil | Soil units, properties, interpretations, and uncertainty. | Reference Soil-owned facts; do not copy them into Agriculture authority. |
| Hydrology | Waterbodies, flows, water-use context, rights/regulatory distinctions. | Preserve ownership; irrigation links do not establish water rights. |
| Atmosphere | Weather, climate, drought, heat, and atmospheric observations/models. | Preserve observed/model/aggregate character and time support. |
| Hazards | Events, warnings, exposure, and resilience context. | Never become alert or emergency authority. |
| Habitat/Flora/Fauna | Habitat, vegetation, taxa, occurrence, sensitivity. | Join only under output-level sensitivity review. |
| People/DNA/Land | Persons, ownership, parcels, consent, sovereignty, private-land facts. | Fail closed; no operator/owner inference by adjacency. |
| Settlements/Infrastructure/Roads | Built systems and transport context. | Minimize operational and vulnerability detail. |
| Evidence/Policy/Release | Support, decisions, review, release, rollback. | Consume refs and obey; never self-approve. |

### Cross-domain invariants

Every relationship or joined derivative must preserve:

1. owning domain for every atomic fact;
2. each input's source role and knowledge character;
3. the most restrictive rights and sensitivity posture;
4. resolvable evidence support for consequential claims;
5. spatial and temporal support;
6. identity and version lineage;
7. correction, withdrawal, and supersession state;
8. release state and public-exposure limitations.

A join can be more sensitive than every individual input.

## 8. Lifecycle and stage contract

The inherited lifecycle is:

```text
RAW -> WORK / QUARANTINE -> PROCESSED -> CATALOG / TRIPLET -> PUBLISHED
```

Promotion is a governed state transition, not a path write.

### Proposed stage grammar

| Stage | Future Agriculture behavior | Required output | Prohibited shortcut |
|---|---|---|---|
| Fixture | Execute deterministic offline cases. | Test result and optional candidate receipt. | No source activation or production claim. |
| RAW read | Read immutable admitted captures. | Input refs and hashes. | No RAW mutation or public interpretation. |
| WORK | Parse, normalize, map, aggregate, derive, and validate candidates. | Candidate plus method/issue records. | No public serving or automatic truth upgrade. |
| QUARANTINE | Hold unresolved rights, identity, schema, sensitivity, scale, or evidence cases. | Reason-coded audit record. | No silent deletion or promotion. |
| PROCESSED | Emit an immutable validated Agriculture version after required gates. | Versioned object refs and validation/receipt refs. | Not automatically cataloged, released, or public. |
| CATALOG/TRIPLET handoff | Prepare discovery or graph candidates. | Provenance-preserving handoff. | Graph/catalog projection cannot replace canonical records. |
| RELEASE handoff | Prepare immutable candidate, blockers, evidence, policy, correction, and rollback refs. | Release-review package. | Pipeline cannot approve release. |
| PUBLISHED | None directly. | Governed released artifact is owned elsewhere. | No direct write to public stores or applications. |

### Finite stage result

A future runner must return a deterministic result such as:

- completed candidate;
- no-op;
- held for missing dependency;
- quarantined;
- abstained;
- denied;
- errored;
- cancelled;
- retryable failure;
- superseded;
- correction-required.

The exact machine vocabulary remains `NEEDS VERIFICATION`; this README does not create an enum.

## 9. Method, aggregation, and determinism

### Method contract

Every transformation must pin:

- method family and version;
- code revision or immutable package digest;
- accepted specification ID/version/digest;
- source and input object versions/digests;
- native and normalized units;
- spatial support, CRS, geography version, and resolution;
- temporal support and all relevant time kinds;
- parameters, thresholds, model/version refs, and random seed where applicable;
- baseline or comparison version;
- no-data, suppression, quality, and uncertainty rules;
- output identity and canonicalization algorithm;
- limitations and denied interpretations.

### Aggregation is not automatic declassification

An aggregation step must distinguish:

```text
computation succeeded
!= privacy threshold satisfied
!= rights allow derivative release
!= source confidentiality cleared
!= re-identification risk acceptable
!= policy approved
!= EvidenceBundle closed
!= release approved
```

A future aggregation process must record:

- input population and selection rule;
- geography/time grouping;
- threshold profile ID and version;
- suppression and complementary suppression;
- missing/no-data behavior;
- small-cell and dominance checks where applicable;
- join-induced re-identification analysis;
- output precision and audience;
- policy and review refs;
- deterministic input/output hashes.

Numeric thresholds are not defined by this README.

### Identity and replay

Until accepted identity profiles exist:

- preserve source IDs and native labels;
- do not equate geometry digest with field identity;
- do not merge farms, fields, parcels, or operators on similarity alone;
- return ambiguity instead of guessing;
- make repeated identical inputs/spec/method produce identical IDs and hashes;
- make changed inputs, parameters, or versions change identity deterministically;
- record no-op behavior;
- make partial failure and retry idempotent.

## 11. Rights, sensitivity, policy, and security

### Agriculture-specific sensitivity

Potentially sensitive classes include:

- exact or reconstructable field geometry;
- operator, farm, person, or parcel linkage;
- ownership or tenancy implications;
- proprietary yield, production, inventory, price, or financial information;
- irrigation, well-to-field, water-use, permit, or operational relationships;
- pesticide, fertilizer, seed, application, management, or treatment records;
- restricted source data and confidential statistical cells;
- critical supply-chain or infrastructure dependencies;
- rare ecological or archaeological context exposed through agricultural joins.

### Default posture

- public exact field/operator/parcel detail is denied;
- aggregation, generalization, redaction, delay, or withholding must be explicit and reviewed;
- unknown rights fail closed;
- the most restrictive applicable source, rights, sensitivity, audience, join, lifecycle, and release rule wins;
- aggregation does not erase source restrictions or re-identification risk;
- generated summaries may not reveal withheld structure;
- logs and receipts must contain references and safe reason codes rather than raw restricted payloads;
- policy must be evaluated through an accepted bundle, input contract, evaluator, and decision vocabulary;
- current Agriculture Rego scaffolds must not be activated until namespace, default, result, tests, and review conflicts are resolved.

### Security requirements

A future implementation must:

- deny secrets in specs, fixtures, logs, receipts, and command arguments;
- run no-network by default;
- cap input size, feature count, geometry complexity, memory, and runtime;
- reject malformed paths, archives, geometries, units, and timestamps;
- minimize exception and validation output;
- pin dependency, schema, policy, and method versions;
- isolate temporary files;
- avoid import-time network or side effects;
- make cancellations and retries leave no ambiguous promoted state.

## 14. Receipts, proofs, and evidence closure

### Separation

```text
Run/Transform/Aggregation receipt  = what process occurred
Validation result                  = what configured checks concluded
PolicyDecision                     = what operation/exposure is permitted
EvidenceBundle                     = what evidence supports a claim
ProofPack                          = assembled proof for a governed transition
Release record                     = what public state may change
```

None substitutes for another.

### Future run receipt obligations

A run should record:

- stable run, pipeline, spec, source, method, and candidate identities;
- code, package, spec, contract, schema, policy, and fixture versions/digests;
- input refs and hashes;
- source roles and knowledge characters;
- geography/time support;
- parameters, seed, resource limits, and environment;
- validation, policy, aggregation, evidence, and review refs;
- output refs/hashes;
- finite result and safe reason codes;
- retry/no-op/cancel/partial-state information;
- correction/supersession dependencies;
- timestamps in UTC;
- no sensitive payloads unless the receipt class and access policy explicitly permit them.

### Evidence rule

Consequential Agriculture claims must resolve `EvidenceRef` to adequate `EvidenceBundle` support or return a bounded negative result. Pipeline execution cannot manufacture evidence from:

- input plausibility;
- schema validity;
- model confidence;
- aggregation;
- a successful join;
- test success;
- a workflow conclusion;
- generated language.

## 15. Promotion, release, correction, and rollback

### Proposed trust chain

```text
admitted immutable input
  -> deterministic WORK candidate or QUARANTINE record
  -> contract/schema/semantic validation
  -> rights/sensitivity/aggregation policy decision
  -> evidence support and review
  -> immutable PROCESSED Agriculture version
  -> catalog/triplet handoff
  -> independent release candidate
  -> ReleaseManifest + correction path + rollback target
  -> governed public-safe carrier
```

Every arrow is conditional.

### Current release posture

The [`release/agriculture/`](../../../release/agriculture/README.md) index reports:

- no concrete Agriculture release approval established;
- the inspected county/year candidate remains proposed;
- the candidate is blocked for evidence and validation;
- the domain workflow contains no accepted release dry-run command.

This pipeline README does not change that state.

### Correction propagation

A mature system must trace each derivative to every source, object, method, policy, evidence, and release dependency. When a dependency is corrected, withdrawn, superseded, reclassified, made more sensitive, or rights-restricted:

1. identify affected candidates, processed versions, catalogs, graphs, releases, caches, exports, and summaries;
2. mark affected outputs stale, held, denied, withdrawn, or under review;
3. stop new public use where required;
4. emit correction/supersession candidates;
5. rebuild only from accepted inputs;
6. preserve history and receipts;
7. expose public correction state through governed carriers when material.

### Rollback

- candidate rollback invalidates or supersedes without deleting process history;
- code/spec rollback restores the prior accepted implementation and disables incompatible schedules/consumers;
- released-artifact rollback belongs to `release/`;
- rollback must not restore a known privacy, rights, sensitivity, evidence, or false-authority defect;
- reverting this README restores documentation only and changes no runtime/public state.

1. Implement one deterministic no-network entrypoint.
2. Bind one accepted spec.

5. Add independently reviewed release dry run.
